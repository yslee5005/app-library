## 1. 실전 프로젝트에서 Claude Code 활용하기

### 1.1 프로젝트 시작: `/init`부터

새 프로젝트에 Claude Code를 처음 도입할 때 가장 먼저 할 일은 `/init`이다.

```bash
cd ~/my-project
claude
> /init
```

`/init`은 코드베이스를 분석해서 CLAUDE.md, 스킬, 훅 설정을 자동 생성해준다. 마치 새 팀원이 왔을 때 "우리 프로젝트는 이렇게 돌아가"라고 온보딩 문서를 만들어주는 셈이다.

> 💡 **초보자 팁**: `/init`이 생성한 CLAUDE.md를 그대로 쓰지 마세요. 꼭 읽어보고 프로젝트에 맞게 수정하세요. 자동 생성은 출발점일 뿐이에요.

### 1.2 일상적인 개발 워크플로우 3가지

**기능 개발**: 계획 세우기 -> 구현 -> 검증 -> `/commit`
```
> "사용자 프로필에 아바타 업로드 기능을 추가하려고 해. 계획 세워줘."
> "계획대로 구현해줘. 테스트도 같이."
> "테스트 돌려보고, 타입 체크도 해줘."
> /commit
```

**버그 수정**: 증상 + 에러 메시지 + 위치 알려주기
```
> "로그인 후 리다이렉트 안 됨. 콘솔에 'undefined redirect URL' 에러.
>  src/auth/ 디렉토리 확인해줘."
```

**리팩토링**: Plan 모드(`/plan`)로 시작 -> 검토 -> 실행
```
> /plan
> "src/utils/ 헬퍼 함수들을 도메인별로 분리하고 싶어. 계획 세워줘."
> "좋아, 이 계획대로 진행해."  (Shift+Tab으로 실행 모드 전환)
```

### 1.3 프롬프트 품질이 결과를 결정한다

택시를 탈 때 "아무데나 가주세요"라고 하면 기사님이 난감한 것처럼, Claude에게도 구체적으로 말해야 한다.

| 나쁜 프롬프트 | 좋은 프롬프트 |
|-------------|------------|
| "로그인 기능 만들어줘" | "이메일/비밀번호 로그인 기능을 src/features/auth/에 만들어줘. Zod 검증, JWT 발급, 401/404 에러 처리, 단위 테스트 포함. 기존 src/middleware/auth.ts 패턴 따라줘." |

> 📌 **핵심 포인트**: 파일 경로, 사용할 라이브러리, 에러 처리 방식, 따라야 할 기존 패턴을 명시하면 결과 품질이 극적으로 올라간다.

### 1.4 점진적 개발: 한 번에 다 하려 하지 마라

복잡한 기능을 한 프롬프트로 만들면, 한 번에 10가지 요리를 시키는 것과 같다. 단계별로 나누자.

```
Step 1: "데이터 모델 먼저. User 스키마에 avatar 필드 추가 + 마이그레이션." -> 확인
Step 2: "파일 업로드 API 엔드포인트 + S3 서비스." -> 확인
Step 3: "프론트엔드 AvatarUpload 컴포넌트." -> 확인
Step 4: "전체 통합 테스트로 검증." -> 완료
```

## 2. CLAUDE.md 잘 쓰는 법

CLAUDE.md는 Claude Code의 "두뇌"다. 잘 쓰면 수년간 함께 일한 동료처럼, 못 쓰면 매번 새 인턴처럼 동작한다.

### 2.1 나쁜 예 vs 좋은 예

**나쁜 예** -- 신입사원에게 "일 잘 해"라고만 말하는 것과 같다:
```markdown
이 프로젝트는 웹 앱입니다. 좋은 코드를 작성해주세요. 테스트를 잘 해주세요.
```

**좋은 예** -- 구체적이고 실행 가능하다:
```markdown
# 프로젝트: MyApp API Server
## 명령어
- 빌드: `pnpm build` / 테스트: `pnpm test` / 린트: `pnpm lint`
## 아키텍처
- 모노레포 (Turborepo), 진입점: `apps/api/src/index.ts`, DB: PostgreSQL + Prisma
## 코딩 규칙
- TypeScript strict, `any` 금지, console.log 금지 (logger.ts 사용)
- 에러 처리: 커스텀 AppError 사용 (src/errors/)
## Git
- 브랜치: `feat/`, `fix/` 접두사, 커밋: Conventional Commits
- PR 전: `pnpm test && pnpm typecheck && pnpm lint` 통과 필수
## 테스트
- Vitest, AAA 패턴, 해당 파일 옆 `__tests__/` 디렉토리
@./docs/api-conventions.md
```

### 2.2 계층별 활용 전략

```
~/.claude/CLAUDE.md (사용자) : 응답 언어, 커밋 스타일, 들여쓰기 등 개인 선호도
프로젝트/CLAUDE.md (팀 공유) : 빌드 명령어, 아키텍처, 코딩 규칙
프로젝트/CLAUDE.local.md     : 내 로컬 DB 포트, 디버깅 노트 (gitignore)
```

### 2.3 모듈화: @include와 Rules

프로젝트가 커지면 `@include`로 모듈화하고, `.claude/rules/`에서 경로별 조건부 규칙을 적용한다.

```markdown
# 루트 CLAUDE.md -- @include로 분산 관리
@./docs/conventions/typescript.md
@./packages/api/CLAUDE.md
@./packages/web/CLAUDE.md
```

```markdown
# .claude/rules/api-rules.md -- src/api/** 작업 시에만 로드
---
paths: ["src/api/**", "src/services/**"]
---
- 모든 API 엔드포인트는 Zod 스키마 검증 필수
- Response DTO는 dto/ 디렉토리에 정의
```

> 📌 **핵심 포인트**: Rules의 경로 타겟팅으로 컨텍스트를 절약하면서 상황에 맞는 규칙만 로드할 수 있다.

## 3. 멀티 에이전트 워크플로우 실전

### 3.1 언제 쓰고, 언제 쓰지 말까

혼자서 이사하는 것과 친구 3명과 이사하는 것의 차이다.

| 멀티 에이전트가 효과적 | 쓰지 말아야 할 때 |
|---------------------|----------------|
| 테스트 + 구현 + 문서화 동시 진행 | 파일 하나 읽기 |
| 여러 마이크로서비스 동시 수정 | 단순 grep 검색 |
| 대규모 리팩토링 병렬 검증 | 작은 버그 수정 |

> 💡 **초보자 팁**: 서브 에이전트는 오버헤드가 크다 (독립 컨텍스트 + 새 API 호출). **5개 이상 도구 호출이 필요한 복합 작업**에만 사용하세요.

### 3.2 실전 사용법

```
"아바타 업로드 기능을 병렬로 진행해줘:
1. 백엔드: POST /api/users/avatar + S3 서비스 (src/api/users/)
2. 프론트: AvatarUpload 컴포넌트 (src/components/profile/)
3. 테스트: 위 두 가지에 대한 테스트
인터페이스 타입은 src/types/user.ts 패턴을 따라줘."
```

### 3.3 핵심 주의사항: 컨텍스트 격리

서브 에이전트는 **부모의 대화를 모른다**. "아까 말한 대로 수정해줘"가 안 통한다. 프롬프트에 파일 경로, 수정 내용, 기대 결과를 **자체 포함적으로** 작성해야 한다.

### 3.4 Worktree 격리

`claude --worktree feat-avatar`로 별도 git worktree에서 작업하면 메인 브랜치를 건드리지 않고 안전하게 개발할 수 있다.

## 4. CI/CD 통합 패턴

### 4.1 기본 원리

`-p` (print) 모드로 비대화형 실행. CI에서 코드 리뷰, 테스트 생성, 문서 업데이트를 자동화한다.

```bash
claude -p "작업 지시" \
  --output-format json \
  --dangerously-skip-permissions \
  --max-budget-usd 2.00 \
  --max-turns 15
```

### 4.2 GitHub Actions: PR 자동 리뷰

```yaml
name: Claude Code Review
on:
  pull_request:
    types: [opened, synchronize]
jobs:
  review:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
        with: { fetch-depth: 0 }
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm install -g @anthropic-ai/claude-code
      - name: Run Review
        env:
          ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
        run: |
          claude -p "이 PR의 diff를 리뷰해줘. 버그, 보안, 성능 이슈 지적.
          $(git diff origin/main...HEAD)" \
            --dangerously-skip-permissions --max-budget-usd 1.00 > review.md
      - name: Post Comment
        uses: actions/github-script@v7
        with:
          script: |
            const review = require('fs').readFileSync('review.md', 'utf8');
            github.rest.issues.createComment({
              issue_number: context.issue.number,
              owner: context.repo.owner, repo: context.repo.repo, body: review
            });
```

### 4.3 구조화된 출력으로 파이프라인 연결

```bash
claude -p "이 PR의 위험도를 평가해줘" \
  --output-format json \
  --json-schema '{"type":"object","properties":{"risk_level":{"type":"string","enum":["low","medium","high"]},"issues":{"type":"array","items":{"type":"string"}}},"required":["risk_level","issues"]}' \
  --dangerously-skip-permissions
```

### 4.4 CI 필수 안전 설정

| 설정 | 이유 |
|------|------|
| `--max-budget-usd` | 비용 폭주 방지 (필수!) |
| `--max-turns` | 무한 루프 방지 |
| `--dangerously-skip-permissions` | Docker/샌드박스 안에서만 |
| `--output-format json` | 파싱 가능한 출력 |
| Secrets로 API 키 관리 | 코드에 키 노출 방지 |

> 📌 **핵심 포인트**: CI에서 `--dangerously-skip-permissions`는 반드시 컨테이너(Docker) 안에서만. 호스트 머신에서 직접 쓰면 보안 위험이 크다.

## 5. 비용 관리와 최적화

### 5.1 비용 구조

```
비용 = (입력 토큰 x 단가) + (출력 토큰 x 단가)

비용을 올리는 것: 긴 대화, 큰 파일 읽기, 불필요한 도구 호출, 서브 에이전트 남용
비용을 줄이는 것: /compact, /clear, 적절한 모델, 구체적 프롬프트
```

### 5.2 핵심 절약 전략 4가지

**모델 매칭**: 작업 복잡도에 따라 모델을 바꾼다.
```
Haiku  : 단순 질문, 포매팅, 간단한 수정 (저렴하고 빠름)
Sonnet : 일반 개발, 버그 수정 (기본 선택)
Opus   : 복잡한 아키텍처, 대규모 리팩토링 (비싸지만 정확)
```

**컨텍스트 관리**: 새 주제는 `/clear`, 긴 대화는 `/compact`로 정리.

**effort 레벨**: `--effort low`(간단 질문), `--effort high`(심층 분석).

**예산 제한**: CI에서는 `--max-budget-usd`와 `--max-turns`를 반드시 설정.

> 🤔 **자주 묻는 질문**: "서브 에이전트를 쓰면 비용이 2배?"
> -> 네, 각 서브 에이전트는 독립 API 호출이므로 비용 추가. 단순 작업엔 직접 도구(Read, Grep)를 쓰세요.

## 6. 흔한 실수와 해결법

### 6.1 "CLAUDE.md 수정했는데 반영이 안 돼요"

**원인**: 세션 시작 시 1회만 읽히고 메모이제이션됨.
**해결**: `/clear`로 새 세션 시작, 또는 `claude --resume <id>`로 재개.

### 6.2 "아까 얘기한 거 기억 안 나?"

**원인**: 컴팩션으로 오래된 메시지가 요약됨.
**해결**: 중요한 맥락은 최근 메시지에서 다시 언급. 핵심 정보는 CLAUDE.md에 기록.

### 6.3 "Claude가 같은 실수를 반복해요"

**원인**: CLAUDE.md에 규칙이 없거나, 프롬프트가 모호함.
**해결**: CLAUDE.md에 "절대 하지 말 것" 섹션 추가. `any` 금지, `console.log` 금지 등.

### 6.4 "권한 확인이 너무 자주 떠요"

**해결**: `--permission-mode acceptEdits`로 파일 편집 자동 승인. 또는 settings.json에 자주 쓰는 명령을 allow 목록에 추가.

```json
{ "permissions": { "allow": ["Bash(npm test)", "Bash(git:*)", "Edit", "Write"] } }
```

### 6.5 "서브 에이전트가 엉뚱한 일을 해요"

**원인**: 프롬프트에 맥락 부족. 서브 에이전트는 부모 대화를 모름.
**해결**: 파일 경로, 기대 결과, 따를 패턴, 결과 크기 제한을 프롬프트에 명시.

### 6.6 "비용이 예상보다 폭발했어요"

**해결**: `--max-budget-usd`와 `--max-turns`를 **항상** 설정. CI에서는 선택이 아니라 필수.

## 7. 프로 팁 모음

### 7.1 생산성 극대화

**`--bare` 모드**: 훅, CLAUDE.md 등을 건너뛰어 즉시 시작. 단발성 질문에 최적.
```bash
claude --bare -p "TypeError: Cannot read property 'x' of null 이거 뭐야?"
```

**Shift+Tab 모드 전환**: Plan 모드(분석만) <-> 실행 모드. 대규모 작업 전 Plan 모드로 먼저 분석하는 습관.

**전략적 /compact**: `"/compact DB 마이그레이션 내용만 유지해줘"`처럼 포커스 지정하면 불필요한 대화가 정리됨.

### 7.2 Hook 자동화 레시피

**자동 포매팅** (파일 저장 시):
```json
{ "hooks": { "PostToolUse": [{ "matcher": "Write", "type": "command",
    "command": "prettier --write $CLAUDE_FILE_PATH 2>/dev/null || true" }] } }
```

**위험 명령어 차단**:
```json
{ "hooks": { "PreToolUse": [{ "matcher": "Bash", "type": "command",
    "command": "echo $CLAUDE_TOOL_INPUT | jq -r '.command' | grep -qE '(rm -rf /|DROP TABLE)' && exit 2 || exit 0" }] } }
```

**Stop Hook으로 테스트 강제**: exit 2를 반환하면 Claude가 멈추지 않고 추가 턴을 받아 문제를 수정한다.
```json
{ "hooks": { "Stop": [{ "type": "command",
    "command": "npm test --silent 2>&1 || (echo '테스트 실패!' >&2 && exit 2)" }] } }
```

### 7.3 Skill로 반복 작업 원클릭화

```markdown
# .claude/skills/new-api/SKILL.md
---
description: 새 API 엔드포인트 생성
argument-hint: endpoint-name
allowed-tools: Write, Bash, Read
---
`$ARGUMENTS` API 엔드포인트를 생성해줘.
1. `src/api/$ARGUMENTS/route.ts` - 라우트 핸들러
2. `src/api/$ARGUMENTS/schema.ts` - Zod 스키마
3. `src/api/$ARGUMENTS/__tests__/route.test.ts` - 테스트
기존 `src/api/health/` 패턴을 따라줘. 완성 후 `pnpm typecheck` 확인.
```
사용: `/new-api user-avatar`

### 7.4 Git 워크플로우

```bash
# 기능별 격리된 워크트리 + tmux (여러 작업 병렬)
claude --worktree feat-avatar --tmux

```

### 7.5 MCP 외부 서비스 연동

```bash
claude mcp add slack -- npx -y @anthropic/mcp-server-slack
claude mcp add postgres -- npx -y @anthropic/mcp-server-postgres \
  --connection-string "postgresql://user:pass@localhost:5432/mydb"
```

MCP를 쓰면 Claude가 직접 DB를 조회하거나 Slack에 메시지를 보내거나 Jira 티켓을 만들 수 있다.

### 7.6 팀 전체 생산성 올리기

```
1. CLAUDE.md를 팀 위키처럼 관리 -> Git 커밋, PR 리뷰에 포함
2. Skills를 팀 공유 자산으로 -> /deploy, /release, /new-component
3. Hook으로 팀 표준 강제 -> 자동 포매팅, 위험 차단, 테스트 강제
4. settings.json 권한 규칙 공유 -> .claude/settings.json을 Git에 커밋
```

### 7.7 보안 체크리스트

```
[ ] .env, credentials -> .gitignore 필수
[ ] CLAUDE.md에 API 키/비밀번호 절대 포함 X
[ ] CI의 --dangerously-skip-permissions -> Docker 안에서만
[ ] deny 규칙으로 위험한 명령어 차단
[ ] --max-budget-usd로 비용 폭주 방지
[ ] MCP 서버 권한 범위 최소화
```
