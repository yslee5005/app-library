# 3. 권한 파이프라인

> 이 문서를 읽으면: Claude Code가 도구 실행 전에 **어떤 기준으로 허용/차단을 결정하는지**, Bash 23단계 보안이 무엇인지 이해할 수 있습니다.

---

## 한 줄 요약

> 모든 도구 호출은 **Allow → Deny → Ask** 순서로 평가된다. Bash는 추가로 23단계 보안 검증을 거친다.

---

## 권한 결정 흐름

```
도구 호출 요청
  │
  ├─① Allow 규칙 매칭? ──── YES → 즉시 실행
  │                         (읽기 전용 도구, 이전에 허용한 패턴)
  │
  ├─② Deny 규칙 매칭? ───── YES → 차단, 모델에 거부 통보
  │                         (사용자가 설정한 금지 규칙)
  │
  └─③ 둘 다 아님 ──────── Ask → 사용자에게 승인 요청
                            ├── 승인 → 실행
                            └── 거부 → 모델에 거부 통보
                                      → 대안 모색
```

---

## 5가지 권한 모드

| 모드 | 설명 | 사용자 확인 빈도 |
|------|------|----------------|
| **Default** | 안전한 읽기는 자동, 나머지는 물어봄 | 중간 |
| **Plan** | 읽기만 가능, 수정/실행 차단 | 없음 (수정 불가) |
| **Auto** | 대부분 자동 승인 (YOLO Classifier) | 최소 |
| **Bypass** | 모든 것 자동 승인 | 없음 |
| **Custom** | 사용자 정의 allow/deny 규칙 | 규칙에 따라 |

### YOLO Classifier (Auto 모드)

유출 소스에서 발견된 자동 분류기:

```
도구 호출
  │
  ├── LOW 리스크  → 자동 승인 (파일 읽기, 검색 등)
  ├── MEDIUM 리스크 → 조건부 승인
  └── HIGH 리스크 → 사용자 확인 필요
```

## 도구별 기본 권한

### 항상 자동 허용 (읽기 전용)

```
Read        파일 읽기
Glob        파일명 검색
Grep        파일 내용 검색
WebSearch   웹 검색
LS          디렉토리 목록
```

왜? → 시스템에 변화를 주지 않으므로 안전.

### 조건부 허용

```
Edit        ─── 허용된 경로 → 자동 / 새 경로 → Ask
Bash        ─── 읽기 전용 명령(ls, cat, git log) → 자동
              ─── 나머지 → 23단계 검증 후 판단
```

### 항상 Ask

```
Write       새 파일 생성
Bash        위험 명령 (rm, curl, sudo 등)
```

## Bash 23단계 보안 파이프라인

`bashSecurity.ts` — **2,500줄**의 보안 검증 코드.

모든 Bash 명령이 거치는 검증:

### 1~18단계: Zsh 빌트인 차단

```
차단 대상: 18개 Zsh 빌트인 명령
  zmodload, emulate, autoload, compdef, ...

이유: Zsh의 특수 기능을 악용한 보안 우회 방지
```

### 19단계: Zsh Equals Expansion 방어

```
공격 벡터:
  =curl  →  Zsh가 이것을 curl 바이너리 경로로 확장
  →  curl에 대한 권한 검사를 우회할 수 있음

방어: = 접두사 패턴 감지 및 차단
```

### 20단계: 유니코드 Zero-Width Space 감지

```
공격 벡터:
  r​m -rf /  →  "rm" 사이에 보이지 않는 유니코드 문자 삽입
  →  사람 눈에는 정상으로 보이지만, 보안 검사 우회

방어: zero-width 문자(U+200B 등) 감지 및 차단
```

### 21단계: IFS Null-Byte 주입 감지

```
공격 벡터:
  IFS=\0 을 조작하여 명령어 파싱을 혼란시킴
  →  의도치 않은 명령 실행 가능

방어: IFS 변수 조작 패턴 감지
```

### 22단계: Malformed 토큰 바이패스 방지

```
출처: HackerOne 보안 리뷰에서 발견
내용: 비정상적 토큰 구조로 보안 검사를 우회하는 기법
방어: 토큰 구조 정규화 및 검증
```

### 23단계: 50 서브커맨드 제한

```
명령어의 파이프라인/서브커맨드가 50개 초과 시:
  → 개별 분석 포기
  → generic "Ask" 프롬프트로 폴백

주의: 이 부분에 CC-643 취약점이 있었음
  → 악성 CLAUDE.md가 50+ 서브커맨드를 생성하도록 유도
  → 보안 검사가 모두 건너뛰어짐
  → Anthropic 내부에서 수정 완료 (유출 시점에 미배포)
```

### 추가 검사 패턴

```
명시적 승인 필요:
  - 명령어 치환: `...`, $(), ${}
  - 프로세스 치환: <(), >()
  - 의심스러운 리다이렉션
  - ANSI-C 인용법: $'...'
  - /proc/*/environ 접근
```

## 권한 규칙 설정 방법

### Allow 규칙

```json
// settings.json
{
  "permissions": {
    "allow": [
      "Edit",                    // Edit 도구 전체 허용
      "Bash(npm test)",          // 특정 Bash 명령만 허용
      "Bash(git *)",             // git 관련 전부 허용
      "mcp__discord__reply"      // 특정 MCP 도구 허용
    ]
  }
}
```

### Deny 규칙

```json
{
  "permissions": {
    "deny": [
      "Bash(rm -rf *)",         // 위험 명령 차단
      "Bash(sudo *)",           // sudo 차단
      "Write(*.env)",           // .env 파일 생성 차단
    ]
  }
}
```

### 우선순위

```
Deny > Allow > Default

1. Deny에 매칭되면 → 무조건 차단
2. Allow에 매칭되면 → 자동 허용
3. 둘 다 아니면 → 모드에 따라 결정
```

## 원격 Killswitch (6개)

유출 소스에서 발견. Anthropic이 **원격으로** 권한을 제어할 수 있는 메커니즘:

```
60분마다 원격 설정 폴링
  │
  ├── 권한 프롬프트 바이패스 (파일/터미널 접근)
  ├── Fast Mode 토글
  ├── Voice Mode 토글
  ├── Analytics 제어
  ├── 강제 종료 (exit)
  └── 피처 플래그 일괄 제어

사용자가 policySettings를 거부하면 → 앱 강제 종료
(one-way enforcement — 사용자 오버라이드 불가)
```
