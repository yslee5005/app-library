# 1. 에이전틱 루프 전체 흐름

> 이 문서를 읽으면: Claude Code가 사용자 메시지를 받고 최종 응답을 내놓기까지 **6단계 루프가 어떻게 도는지** 완전히 이해할 수 있습니다.

---

## 한 줄 요약

> 사용자가 말하면 → Claude가 생각하고 → 도구를 쓰고 → 결과를 보고 → 또 생각하고 → 더 이상 도구가 필요 없을 때까지 반복

---

## 챗봇과 에이전트의 근본적 차이

| | 일반 챗봇 | Claude Code (에이전트) |
|--|----------|----------------------|
| 패턴 | 질문 → 답변 (1회) | 질문 → 도구 → 관찰 → 도구 → ... → 답변 (N회) |
| 코드 | 코드를 **텍스트로 보여줌** | 코드를 **직접 수정**함 |
| 환경 | 격리된 서버 | **당신의 로컬 머신** |
| 종료 조건 | 항상 1회 응답 | 도구 호출이 없을 때 종료 |

**핵심**: 일반 챗봇은 "말"만 하지만, 에이전트는 "행동"한다.

---

## 6단계 순환 구조

```
   ┌──① 사용자 메시지─────────────────────────┐
   │                                           │
   │  ② 컨텍스트 조립                           │
   │     CLAUDE.md + 대화 기록 + 이전 도구 결과    │
   │     + 환경변수 + MCP 리소스                  │
   │                                           │
   │  ③ Claude API에 전송                       │
   │     → 모델이 "다음에 뭘 할지" 판단            │
   │     → 텍스트 응답 or 도구 호출 결정           │
   │                                           │
   │  ④ 권한 확인                               │
   │     → Allow: 자동 진행                      │
   │     → Ask: 사용자에게 물어봄                  │
   │     → Deny: 차단                           │
   │                                           │
   │  ⑤ 도구 실행                               │
   │     → 파일 읽기, Bash 명령, 검색, 웹 등       │
   │     → 결과를 구조화된 형태로 반환              │
   │                                           │
   │  ⑥ 루프 판단                               │
   │     ├─ 도구 호출 더 있음 → ②로 돌아감         │
   │     └─ 도구 호출 없음   → 최종 응답 출력       │
   └───────────────────────────────────────────┘
```

---

## 각 단계 상세

### ① 사용자 메시지

터미널에 입력하는 모든 것. 예:

```
"src/auth.ts에서 로그인 버그 찾아서 고쳐줘"
```

### ② 컨텍스트 조립

Claude가 "생각"하기 전에, 필요한 정보를 모아서 하나의 프롬프트로 조립:

```
조립되는 것들:
  ├── 시스템 프롬프트 (Claude Code의 행동 규칙)
  ├── CLAUDE.md (프로젝트/사용자 지시사항)
  ├── 대화 기록 (이전 메시지들)
  ├── 이전 도구 실행 결과
  ├── 환경 정보 (OS, shell, git 상태 등)
  └── MCP 리소스 (연결된 외부 서비스 정보)
```

**중요**: 이 단계에서 컨텍스트가 창 한계에 가까우면 **자동 압축**이 발동됨 (→ [04-컨텍스트-압축.md](./04-컨텍스트-압축.md) 참조)

### ③ Claude API 전송 & 판단

조립된 컨텍스트가 Claude API로 전송되면, 모델이 두 가지 중 하나를 결정:

```
Claude의 판단:
  ├── "도구를 써야겠다" → 도구 호출 생성 (④로 진행)
  └── "답변으로 충분하다" → 텍스트 응답 (루프 종료)
```

Claude는 **여러 도구를 동시에 호출**할 수 있음:
```
Claude: "이 두 파일을 동시에 읽어야겠다"
  → Read(src/auth.ts) + Read(src/middleware.ts)  ← 병렬 실행
```

### ④ 권한 확인

도구 호출이 결정되면, 실행 전에 **권한 파이프라인** 통과:

```
도구 호출 요청
  │
  ├── Allow 규칙 매칭? → 바로 실행
  │     (Read, Glob 등 안전한 도구)
  │
  ├── Deny 규칙 매칭? → 차단, 모델에 거부 통보
  │
  └── 그 외 → Ask → 사용자에게 승인 요청
        ├── 승인 → 실행
        └── 거부 → 모델에 거부 통보, 대안 모색
```

상세한 권한 시스템은 → [03-권한-파이프라인.md](./03-권한-파이프라인.md) 참조

### ⑤ 도구 실행

승인된 도구가 **격리된 컨텍스트**에서 실행:

```
도구 실행
  ├── 스키마 검증 (Zod) → 입력이 올바른지
  ├── 샌드박스 실행 → 다른 도구와 격리
  ├── 결과 생성 → 구조화된 출력
  └── 결과 반환 → 다음 루프의 컨텍스트에 추가
```

### ⑥ 루프 판단

도구 실행 결과가 돌아오면:

- **모델이 추가 도구 호출을 요청** → ②로 돌아감 (루프 계속)
- **모델이 텍스트만 응답** → 루프 종료, 사용자에게 결과 출력

---

## 실전 예시: "버그 찾아서 고쳐줘"

```
사용자: "src/auth.ts에서 로그인 버그 찾아서 고쳐줘"

── 루프 1 ──
② 컨텍스트 조립
③ Claude: "먼저 파일을 읽어야겠다"
④ Read → 자동 허용 (읽기 전용)
⑤ Read(src/auth.ts) 실행 → 파일 내용 반환
⑥ 도구 호출 더 있음 → 루프 계속

── 루프 2 ──
② 파일 내용이 컨텍스트에 추가됨
③ Claude: "23번줄에 off-by-one 버그 발견. 수정하겠다"
④ Edit → Ask → 사용자 승인
⑤ Edit(line 23 수정) 실행 → 수정 완료
⑥ 도구 호출 더 있음 → 루프 계속

── 루프 3 ──
② 수정 결과가 컨텍스트에 추가됨
③ Claude: "테스트를 돌려서 확인하겠다"
④ Bash(npm test) → Ask → 사용자 승인
⑤ npm test 실행 → 테스트 통과
⑥ 도구 호출 없음 → 루프 종료

최종 응답: "23번줄 off-by-one 버그를 수정했고, 테스트 통과했습니다"
```

루프를 **3바퀴** 돌았습니다.

## 전부 로컬에서 실행된다

```
┌──────────────────────────────┐
│      당신의 터미널 프로세스       │
│                              │
│  Claude Code 프로세스          │
│    ├── 에이전틱 루프            │
│    ├── 도구 실행               │
│    ├── 파일 읽기/쓰기          │
│    └── Bash 명령 실행          │
│                              │
│  유일하게 외부 통신:            │
│    └── Claude API 호출         │
│        (당신의 메시지 → 모델 응답) │
└──────────────────────────────┘
```

- 원격 실행 서버 **없음**
- 파일 접근, Bash 실행 모두 **로컬**
- API 호출만 네트워크 사용

## 유출 소스에서 밝혀진 내부 수치

| 항목 | 수치 |
|------|------|
| 루프를 제어하는 QueryEngine.ts | **46,000줄** |
| 루프 안에서 사용 가능한 도구 | **46+개** |
| 프롬프트 캐시 break vector 추적 | **14개** |
| 컨텍스트 한계 도달 시 | **자동 압축** (4단계) |
| 압축 실패 시 서킷 브레이커 | **3회 연속 실패** |
| 패치 전 일일 낭비 API 호출 | **~250,000회** |
