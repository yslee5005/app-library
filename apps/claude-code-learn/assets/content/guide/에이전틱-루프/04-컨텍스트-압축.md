# 4. 컨텍스트 압축

> 이 문서를 읽으면: 대화가 길어져서 컨텍스트 창이 꽉 차면 Claude Code가 **어떻게 정보를 정리하고 압축하는지** 이해할 수 있습니다.

---

## 한 줄 요약

> 4단계 압축 파이프라인: **도구 결과 트리밍 → MicroCompact → AutoCompact → Full Compact** 순서로 점점 더 강하게 압축

---

## 왜 압축이 필요한가?

```
에이전틱 루프가 10바퀴 돌면:
  → 도구 결과 10개가 컨텍스트에 쌓임
  → 파일 내용, Bash 출력, 검색 결과...
  → 컨텍스트 창(1M 토큰)도 결국 한계

압축 없이 계속 쌓으면:
  ① 컨텍스트 창 초과 → API 에러
  ② 토큰 비용 폭증
  ③ 오래된 정보가 최신 정보를 밀어냄
```

---

## 4단계 압축 파이프라인

```
대화가 진행됨
  │
  ├─① 도구 결과 예산(budgeting)
  │    → 오래된 도구 출력을 시간 기반으로 트리밍
  │
  ├─② MicroCompact
  │    → 캐시된 콘텐츠를 로컬에서 편집 (API 호출 0)
  │
  ├─③ AutoCompact
  │    → 컨텍스트 창 상한 근접 시 자동 발동
  │    → Claude API로 구조화된 요약 생성
  │
  └─④ Full Compact (/compact 명령)
       → 전체 대화 압축 + 선택적 재주입
```

각 단계는 **이전 단계로 충분하지 않을 때**만 다음으로 넘어감.

---

## ① 도구 결과 트리밍

가장 가벼운 단계. 시간이 지난 도구 출력을 줄임:

```
트리밍 대상:
  ✅ COMPACTABLE_TOOLS 목록에 있는 도구
     (Read, Edit, Bash, Grep, Glob 등)

트리밍 제외:
  ❌ MCP 도구 결과 → 항상 보존
  ❌ Agent 도구 결과 → 항상 보존
  ❌ 커스텀 도구 결과 → 항상 보존
```

왜 MCP는 제외? → 외부 서비스 결과는 다시 가져오기 어려우므로 보존.

---

## ② MicroCompact

**API 호출 없이** 로컬에서 처리:

```
동작 방식:
  - 오래된 도구 출력을 직접 편집
  - 캐시된 콘텐츠를 축약
  - Claude API 호출 0회 (비용 0)

예시:
  Before: Read(src/auth.ts) → 500줄 전체 내용
  After:  Read(src/auth.ts) → "[파일 내용 생략 - 500줄]"
```

가장 비용 효율적인 압축.

---

## ③ AutoCompact

컨텍스트 창 한계에 가까워지면 **자동 발동**:

```
트리거 조건:
  컨텍스트 사용량 > (컨텍스트 창 - 13,000 토큰 버퍼)

동작:
  1. Claude API에 "이 대화를 요약해줘" 요청
  2. 최대 20,000 토큰의 구조화된 요약 생성
  3. 원본 대화를 요약으로 교체

서킷 브레이커:
  3회 연속 실패 시 → 무한 재시도 대신 중단
  (패치 전에는 이것 때문에 일일 ~250,000 API 호출 낭비)
```

### AutoCompact가 보존하는 것

```
우선순위 높음 (보존):
  ├── 최근 사용자 메시지
  ├── 현재 작업 컨텍스트
  ├── MCP 도구 결과
  └── 중요한 결정 사항

우선순위 낮음 (제거 대상):
  ├── 오래된 도구 출력
  ├── 중간 과정 출력
  └── 탐색적 시도 (실패한 접근법)
```

---

## ④ Full Compact (/compact 명령)

사용자가 수동으로 트리거하거나, AutoCompact로도 부족할 때:

```
/compact 실행 시:

1. 전체 대화 기록 압축
2. 핵심 사실만 추출하여 요약
3. 작업 예산: 50,000 토큰
4. 파일별 재주입 제한: 5,000 토큰
5. 압축된 컨텍스트로 세션 계속
```

---

## 5가지 압축 전략

내부적으로 5가지 전략을 상황에 따라 선택:

| 전략 | 설명 | 사용 시점 |
|------|------|----------|
| **시간 기반 제거** | 오래된 도구 결과 제거 | 가장 먼저 시도 |
| **대화 요약** | 대화 흐름을 요약으로 교체 | 중간 단계 |
| **세션 메모리 추출** | 중요 사실을 메모리로 이동 | 장기 세션 |
| **전체 히스토리 요약** | 모든 것을 하나의 요약으로 | 거의 한계 |
| **가장 오래된 메시지 절삭** | 최후의 수단 | 한계 초과 |

---

## 메모리와의 관계

압축은 **정보를 버리는 것**이 아니라 **계층을 이동시키는 것**:

```
컨텍스트 창 (휘발성, 빠름)
  │ 압축 시 중요 정보 추출
  ▼
세션 메모리 (반영구, 현재 세션)
  │ 세션 종료 시
  ▼
MEMORY.md (영구, 200줄 인덱스)
  │ 상세 내용 참조
  ▼
토픽 파일 (영구, 온디맨드 로딩)
```

---

## 실전 팁

### 컨텍스트가 비효율적으로 쓰이는 경우

```
❌ 큰 파일을 통째로 Read
   → 필요한 부분만 offset/limit으로 읽기

❌ Bash 출력을 그대로 두기
   → 긴 출력은 자연스럽게 트리밍됨

❌ 같은 파일을 반복해서 Read
   → 파일 미변경 시 캐시된 스텁 반환 (자동 최적화)
```

### /compact를 써야 할 때

```
✅ 세션이 30분 이상 지속되었을 때
✅ "컨텍스트 한계" 경고가 나올 때
✅ Claude가 이전 맥락을 잊어버리는 것 같을 때
✅ 작업 방향이 크게 바뀌었을 때 (이전 맥락 불필요)
```
