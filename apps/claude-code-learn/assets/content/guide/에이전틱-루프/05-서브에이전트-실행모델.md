# 5. 서브 에이전트 실행 모델

> 이 문서를 읽으면: Claude Code가 복잡한 작업을 **어떻게 쪼개서 병렬로 처리하는지**, 3가지 서브 에이전트 모델의 차이를 이해할 수 있습니다.

---

## 한 줄 요약

> 서브 에이전트는 **Fork / Teammate / Worktree** 3가지 모델로 실행되며, 각각 격리 수준과 용도가 다르다.

---

## 왜 서브 에이전트가 필요한가?

```
단일 에이전트의 한계:
  - 한 번에 하나씩 순차 실행 → 느림
  - 긴 작업이 메인 컨텍스트를 오염
  - 위험한 작업이 안전한 작업에 영향

서브 에이전트로 해결:
  - 독립 작업을 병렬 실행 → 빠름
  - 각자의 컨텍스트에서 격리 실행
  - 결과만 부모에게 반환 (중간 과정 X)
```

---

## 핵심 설계 원칙

### 서브 에이전트 = 도구의 일종

```
일반 도구:     Read, Edit, Bash
에이전트 도구:  Agent (서브 에이전트 생성)

→ 같은 팩토리, 같은 권한 관리, 같은 격리
→ 특별 취급이 아니라 일급 시민(first-class citizen)
```

### 결과만 반환 (Output-Only)

```
부모 에이전트
  │
  ├── 서브 에이전트 A 생성
  │     ├── 자체 컨텍스트에서 작업
  │     ├── Read 5번, Grep 3번, Bash 2번 실행
  │     └── 최종 결과만 반환 ─────────┐
  │                                   │
  └── 서브 에이전트 B 생성              │
        ├── 자체 컨텍스트에서 작업       │
        ├── Read 3번, Edit 1번 실행    │
        └── 최종 결과만 반환 ──────────┤
                                      │
  부모가 결과 종합 ←──────────────────┘
```

**중요**: 서브 에이전트의 **전체 작업 히스토리는 반환되지 않음**. 결과만.

→ 이유: "위험한 작업이 메인 에이전트의 상태를 오염시키지 않도록"

---

## 3가지 실행 모델

### 모델 1: Fork

```
┌──────────────────────┐
│    부모 에이전트 컨텍스트  │
│    (1000 토큰)         │
│                        │
│  "Fork 생성"           │
│    │                   │
│    ▼                   │
│  ┌──────────────────┐ │
│  │ Fork 에이전트      │ │
│  │ (부모 컨텍스트 복사) │ │
│  │ = 1000 토큰       │ │
│  │                   │ │
│  │ 프롬프트 캐시 적중! │ │
│  └──────────────────┘ │
└──────────────────────┘
```

| 항목 | 설명 |
|------|------|
| **격리 수준** | 낮음 (부모 컨텍스트 복사) |
| **속도** | 가장 빠름 (캐시 적중) |
| **용도** | 빠른 병렬 조회, 간단한 질문 |
| **비용** | 최저 (캐시 재활용) |

**실전 예시**:
```
"이 3개 파일에서 각각 TODO를 찾아줘"

→ Fork 1: Grep(src/auth.ts, "TODO")
→ Fork 2: Grep(src/api.ts, "TODO")
→ Fork 3: Grep(src/db.ts, "TODO")
→ 3개 동시 실행, 결과 합쳐서 보고
```

---

### 모델 2: Teammate

```
┌──────────────────────┐    ┌──────────────────────┐
│    부모 에이전트        │    │  Teammate 에이전트     │
│    (메인 tmux 패인)    │    │  (별도 tmux 패인)      │
│                       │    │                       │
│                       │    │  독자적 컨텍스트       │
│    파일 기반 메일박스   │◄──►│  장시간 독립 실행      │
│    (통신 채널)         │    │  자체 도구 세트        │
│                       │    │                       │
└──────────────────────┘    └──────────────────────┘
```

| 항목 | 설명 |
|------|------|
| **격리 수준** | 중간 (별도 패인, 파일 통신) |
| **속도** | 중간 |
| **용도** | 장시간 독립 작업 |
| **통신** | 파일 기반 메일박스 |

**실전 예시**:
```
"백그라운드에서 전체 테스트 스위트 돌리면서,
 나는 다른 작업 계속할게"

→ Teammate: npm test 실행 (장시간)
→ 부모: 다른 파일 편집 계속
→ Teammate 완료 시 메일박스로 결과 전달
```

---

### 모델 3: Worktree

```
┌──────────────────────┐    ┌──────────────────────┐
│  메인 브랜치 (main)    │    │  Worktree 브랜치      │
│                       │    │  (agent/task-xyz)     │
│  부모 에이전트가        │    │                       │
│  안전하게 작업 중       │    │  Worktree 에이전트     │
│                       │    │  완전 격리된 git 환경   │
│  git 상태 영향 없음    │    │  위험한 변경 가능      │
│                       │    │                       │
└──────────────────────┘    └──────────────────────┘
```

| 항목 | 설명 |
|------|------|
| **격리 수준** | 최고 (별도 git worktree + 브랜치) |
| **속도** | 가장 느림 (worktree 생성 오버헤드) |
| **용도** | 위험한 코드 변경, 실험적 리팩토링 |
| **안전성** | 실패해도 메인 브랜치 영향 없음 |

**실전 예시**:
```
"이 함수를 완전히 다시 작성해봐.
 근데 실패하면 원래대로 돌아갈 수 있어야 해"

→ Worktree 생성 (별도 브랜치)
→ Worktree 에이전트: 함수 재작성 + 테스트
  ├── 성공 → 부모에게 diff 반환, 머지 가능
  └── 실패 → worktree 삭제, 메인 영향 없음
```

---

## 비교 표

| | Fork | Teammate | Worktree |
|--|------|----------|----------|
| **격리** | 컨텍스트만 분리 | 프로세스 분리 | git 환경 분리 |
| **속도** | 빠름 | 중간 | 느림 |
| **비용** | 캐시 재활용 | 새 컨텍스트 | 새 컨텍스트 + git |
| **통신** | 결과 반환 | 파일 메일박스 | 결과 반환 |
| **실패 영향** | 메인 컨텍스트 무관 | 메인 컨텍스트 무관 | 메인 git 무관 |
| **용도** | 병렬 조회 | 장시간 작업 | 위험한 변경 |

---

## 멀티 에이전트 오케스트레이션

유출 소스에서 발견된 **COORDINATOR_MODE** (미출시 기능):

```
Coordinator (오케스트레이터)
  │
  ├── Phase 1: Research
  │     ├── Worker A: 코드 분석
  │     └── Worker B: 테스트 분석
  │
  ├── Phase 2: Spec
  │     └── Worker C: 변경 사양 작성
  │
  ├── Phase 3: Implement
  │     ├── Worker D: src/ 수정
  │     └── Worker E: test/ 수정
  │
  └── Phase 4: Verify
        └── Worker F: 전체 테스트 실행

통신 프로토콜: XML 기반 태스크 알림
워커 환경: 각자 격리된 스크래치 디렉토리
충돌 해결: Coordinator가 cross-agent 충돌 해소
```

## 실전 사용 팁

### 언제 어떤 모델을 쓸까

```
"여러 파일에서 뭔가 찾아줘"     → Fork (빠른 병렬 검색)
"테스트 돌리면서 다른 것도 해줘"  → Teammate (장시간 백그라운드)
"큰 리팩토링 시도해봐"          → Worktree (안전한 실험)
"이건 간단하니까 직접 해줘"      → 서브 에이전트 불필요 (직접 실행)
```

### Agent 도구 호출 예시

```
병렬 실행:
  Agent({ description: "Run tests", prompt: "..." })
  Agent({ description: "Check types", prompt: "..." })
  → 두 개 동시에 실행됨

Worktree 격리:
  Agent({
    description: "Refactor auth",
    prompt: "...",
    isolation: "worktree"
  })
  → 별도 git worktree에서 실행

백그라운드:
  Agent({
    description: "Long analysis",
    prompt: "...",
    run_in_background: true
  })
  → 완료 시 알림
```

### 효과적인 프롬프트

```
❌ "이것 좀 조사해봐"
✅ "src/auth.ts의 login() 함수에서 세션 토큰이
    만료되지 않는 버그를 찾아서, 원인과 수정 방안을
    200단어 이내로 보고해줘"

핵심:
  - 목표 명확히
  - 범위 구체적으로
  - 결과 형식 지정
```

## 수치 요약

| 항목 | 수치 |
|------|------|
| 서브 에이전트 실행 모델 | **3종** (Fork, Teammate, Worktree) |
| COORDINATOR_MODE 페이즈 | **4단계** (research, spec, implement, verify) |
| 워커 간 통신 | **XML 프로토콜** |
| 서브 에이전트 결과 | **output-only** (전체 히스토리 X) |
