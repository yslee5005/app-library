# 2. 도구 시스템 패턴

> 이 문서를 읽으면: Claude Code의 46+개 도구가 **어떤 규칙과 패턴으로 정의되는지**, 왜 29,000줄이나 되는지 이해할 수 있습니다.

---

## 한 줄 요약

> 모든 도구는 **하나의 공통 팩토리**를 통해 생성된다. 새 도구를 만들어도 보안 검사를 우회할 수 없는 구조.

---

## 핵심 원칙: Tool Factory 패턴

Claude Code의 도구 시스템은 **플러그인 아키텍처**다. 각 도구(Read, Edit, Bash, Agent...)는 독립 모듈이지만, 모두 같은 팩토리를 거쳐 등록된다.

```
┌──────────────────────────────────────────────┐
│              Tool Factory (공통)               │
│                                               │
│  ┌─────────┐  ┌─────────┐  ┌─────────┐      │
│  │  Read    │  │  Edit   │  │  Bash   │ ...  │
│  │  도구    │  │  도구    │  │  도구   │      │
│  └────┬────┘  └────┬────┘  └────┬────┘      │
│       │            │            │             │
│       └────────────┼────────────┘             │
│                    │                          │
│            ┌───────▼───────┐                  │
│            │  공통 파이프라인  │                  │
│            │               │                  │
│            │ ① Zod 스키마 검증│                  │
│            │ ② 권한 게이트   │                  │
│            │ ③ 격리 실행    │                  │
│            │ ④ 출력 변환    │                  │
│            │ ⑤ 에러 핸들링   │                  │
│            │ ⑥ 텔레메트리    │                  │
│            └───────────────┘                  │
└──────────────────────────────────────────────┘
```

개별 도구는 **"나는 무엇을 하는 도구인가"만 정의**하면 되고, 보안/검증/격리는 팩토리가 자동으로 감싼다.

---

## 도구 정의의 4가지 필수 요소

어떤 도구든 반드시 이 4가지를 선언해야 한다:

### ① 입력 스키마 (Zod)

모든 도구는 **Zod v4**로 입력을 정의:

```typescript
// Read 도구 예시
const ReadSchema = z.object({
  file_path: z.string(),          // 필수 - 절대 경로
  offset: z.number().optional(),  // 선택 - 시작 줄
  limit: z.number().optional(),   // 선택 - 최대 줄 수
  pages: z.string().optional(),   // 선택 - PDF 페이지
})

// Edit 도구 예시
const EditSchema = z.object({
  file_path: z.string(),          // 필수
  old_string: z.string(),         // 필수 - 찾을 문자열
  new_string: z.string(),         // 필수 - 바꿀 문자열
  replace_all: z.boolean().default(false),
})

// Bash 도구 예시
const BashSchema = z.object({
  command: z.string(),            // 필수
  timeout: z.number().optional(), // 최대 600,000ms
  description: z.string().optional(),
  run_in_background: z.boolean().optional(),
})
```

**규칙**: 스키마 검증은 API 레이어뿐 아니라 **모든 모듈 경계**에서 실행됨. 잘못된 입력은 도구 실행 전에 차단.

### ② 권한 레벨

각 도구는 **얼마나 위험한지** 스스로 선언:

| 레벨 | 의미 | 해당 도구 |
|------|------|----------|
| **자동 허용** | 위험 없음, 물어보지 않음 | Read, Glob, Grep, WebSearch |
| **조건부 허용** | 특정 조건에서만 자동 | Bash (읽기 전용 명령), Edit (허용된 경로) |
| **Ask** | 사용자 확인 필요 | Edit (새 경로), Write, Bash (위험 명령) |
| **Deny** | 차단 | deny 규칙에 매칭된 모든 도구 |

```
Read     → 읽기 전용   → 항상 자동 허용
Glob     → 파일 검색   → 항상 자동 허용
Edit     → 파일 수정   → 처음은 Ask, 이후 패턴 기억
Write    → 파일 생성   → Ask
Bash     → 만능       → 명령어별 23단계 검사 후 판단
Agent    → 서브 에이전트 → 부모와 동일한 권한 관리
```

### ③ 실행 로직

각 도구는 **격리된 실행 컨텍스트**에서 동작:

```
도구 A 실행 중             도구 B 실행 중
┌──────────────┐         ┌──────────────┐
│ 내부 상태 A   │  ←X→   │ 내부 상태 B   │
│ 파일 핸들 A   │  접근   │ 파일 핸들 B   │
│ 캐시 A       │  불가   │ 캐시 B       │
└──────────────┘         └──────────────┘
```

- 도구 간 **상태 누출(state leakage) 불가능**
- 한 도구의 버그가 다른 도구에 영향 X
- 고위험 도구(Bash)는 **추가 샌드박스**

### ④ 구조화된 출력

모든 도구는 정해진 형태로 결과 반환:

```
도구 실행 결과
  ├── 성공 시: 구조화된 데이터 (파일 내용, 검색 결과 등)
  └── 실패 시: 에러 메시지 + 원인

→ 이 결과가 다음 루프에서 컨텍스트에 추가됨
→ Claude가 결과를 보고 다음 행동 결정
```

## 도구 카테고리별 상세

### 파일 도구 (Read / Edit / Write)

```
공통 특징:
  - file_path는 반드시 절대 경로
  - Edit/Write 전에 반드시 Read 선행
  - 파일 미변경 시 캐시된 스텁 반환 (불필요한 재읽기 방지)

Read의 특수 기능:
  - 이미지(PNG, JPG) → 멀티모달로 "보여줌"
  - PDF → 페이지 단위 읽기 (최대 20페이지)
  - Jupyter 노트북 → 셀 + 출력 + 시각화 반환

Edit의 안전장치:
  - 정규식 아님 → 순수 문자열 비교 (의도치 않은 변경 방지)
  - 여러 매칭 시 실패 → replace_all: true 명시 필요
  - 줄 번호 접두사 자동 제거 (Read 출력 복붙 가능)
```

### 검색 도구 (Glob / Grep)

```
Glob: 파일명 패턴 검색
  - "**/*.ts" → 모든 TypeScript 파일
  - 수정 시간 기준 정렬
  - 용도: "이 프로젝트에 어떤 파일이 있지?"

Grep: 파일 내용 검색 (Ripgrep 기반)
  - 정규식 지원
  - 3가지 출력 모드: files_with_matches, content, count
  - 멀티라인 패턴 지원
  - 용도: "이 함수를 어디서 호출하지?"
```

### Bash 도구

가장 강력하고 가장 위험한 도구:

```
할 수 있는 것: 거의 모든 것
  - 빌드 (npm build, flutter build)
  - 테스트 (npm test, flutter test)
  - Git (commit, push, branch)
  - 패키지 관리 (npm install, pub add)
  - 시스템 명령 (ls, curl, docker)

보안: 23단계 검증 파이프라인
  → 상세 내용은 03-권한-파이프라인.md 참조
```

### Agent 도구

**서브 에이전트도 도구의 하나**로 취급됨:

```
일반적 접근:  서브 에이전트 = 별도 프로세스
Claude Code:  서브 에이전트 = Tool의 일종

→ 같은 권한 관리, 같은 라이프사이클, 같은 격리
→ ad-hoc 프로세스 스포닝이 아니라 제어된 시스템
```

3가지 실행 모델:

| 모델 | 설명 | 용도 |
|------|------|------|
| **Fork** | 부모 컨텍스트 복사, 프롬프트 캐시 적중 | 빠른 병렬 조회 |
| **Teammate** | tmux 별도 패인, 파일 기반 메일박스 | 장시간 독립 작업 |
| **Worktree** | 자체 git worktree, 격리 브랜치 | 위험한 코드 변경 |

### MCP 도구

외부 서비스 연동:

```
네이밍: mcp__{서버명}__{도구명}
  예: mcp__discord__reply
      mcp__linear__create_issue
      mcp__puppeteer__navigate

특징:
  - 동일한 권한 파이프라인 통과
  - 결과는 자동 압축 대상에서 제외 (항상 보존)
  - deny 규칙으로 차단 가능
```

### 웹 도구

```
WebSearch: 웹 검색
  - 제목, URL, 스니펫 포함 결과
  - 내부 비용: $0.01/쿼리

WebFetch: URL 내용 가져오기
  - HTML → Markdown 자동 변환
  - 15분 캐시
  - HTTP → HTTPS 자동 업그레이드
```

## 프롬프트 캐시와 도구 설계의 관계

유출 소스에서 발견된 중요한 설계 원칙:

```
도구 정의가 변경되면
  → 시스템 프롬프트 변경
  → 프롬프트 캐시 무효화
  → 모든 사용자의 API 비용 증가

따라서:
  ✅ 도구 스키마는 최대한 안정적으로 유지
  ✅ 캐시 인식 경계(cache-aware boundary)로 프롬프트 구성
  ❌ 도구 정의를 자주 바꾸지 않음
```

**14개 캐시 break vector**가 추적됨 — 도구 변경이 캐시에 미치는 영향을 모니터링.
