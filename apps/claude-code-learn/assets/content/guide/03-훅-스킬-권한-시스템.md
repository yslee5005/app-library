## 3.1 이 챕터의 큰 그림

Claude Code를 단순한 "대화형 코딩 도우미"에서 **나만의 자동화 시스템**으로 업그레이드하는 세 가지 핵심 개념이 있습니다.

```
┌─────────────────────────────────────────────────────┐
│                  Claude Code                         │
│                                                      │
│   ┌──────────┐  ┌──────────┐  ┌──────────────────┐  │
│   │   훅     │  │   스킬    │  │   MCP 서버       │  │
│   │ (자동화) │  │ (워크플로)│  │ (외부 연결)      │  │
│   └────┬─────┘  └────┬─────┘  └────────┬─────────┘  │
│        │             │                  │            │
│        └─────────────┼──────────────────┘            │
│                      │                               │
│              ┌───────┴───────┐                       │
│              │   권한 시스템   │                       │
│              │  (안전장치)    │                       │
│              └───────────────┘                       │
└─────────────────────────────────────────────────────┘
```

**일상 비유로 이해하기:**

| 개념 | 일상 비유 | 한마디 설명 |
|------|----------|------------|
| 훅 (Hooks) | 스마트홈 자동화 규칙 | "문이 열리면 자동으로 불 켜줘" |
| 스킬 (Skills) | 레시피 카드 | "이 레시피대로 요리해줘" |
| MCP 서버 | USB 허브 | "외부 장치를 연결해서 사용해" |
| 권한 시스템 | 건물 출입 카드 | "이 구역은 허가 없이 못 들어가" |

## 3.2 훅(Hooks) -- 자동으로 움직이는 비서

### 개념 설명

훅은 **"특정 사건이 일어나면 자동으로 무언가를 실행하라"**는 규칙입니다.

스마트홈을 생각해보세요. "현관문이 열리면 거실 불을 켜라", "밤 11시가 되면 모든 불을 꺼라" 같은 자동화 규칙을 설정하죠? Claude Code의 훅도 정확히 같습니다.

```
일상 비유:                          Claude Code 훅:

현관문이 열리면 → 불을 켜라       파일을 저장하면 → 포매팅 실행
밤 11시가 되면 → 불을 꺼라        세션 시작 시 → 환경 설정 로드
아이가 귀가하면 → 알림 보내라     위험한 명령 시 → 차단하라
```

### 왜 중요한가

매번 "코드 저장 후 prettier 실행해줘"라고 말할 필요가 없습니다. 한 번 설정하면 **자동으로** 실행됩니다. 실수로 위험한 명령을 내려도 훅이 **자동으로 차단**합니다. 개발 워크플로우의 일관성을 보장합니다.

### 훅의 동작 원리: 종료 코드

모든 훅의 핵심은 **종료 코드(exit code)** 세 가지입니다.

```
  ┌─────────────┐
  │  훅 실행됨   │
  └──────┬──────┘
         │
    ┌────┴────┐
    │ 종료코드? │
    └────┬────┘
         │
    ┌────┼────────────────┐
    │    │                │
    v    v                v
  [0]  [2]             [기타]
 성공   차단/주입        오류
  │     │                │
  v     v                v
 계속   Claude에게      사용자에게
 진행   알려줌          표시만 함
        (대응 가능)     (실행 계속)
```

> 💡 **초보자 팁**: 종료 코드 `2`는 단순한 에러가 아닙니다. "Claude야, 이건 안 돼. 다른 방법을 찾아봐"라고 말하는 신호입니다. Claude가 이 피드백을 받고 스스로 대안을 찾을 수 있습니다.

### 4가지 훅 타입

훅은 네 가지 방법으로 실행할 수 있습니다.

#### 타입 1: 셸 명령어 (가장 많이 쓰임)

터미널 명령어를 직접 실행합니다. 포매팅, 테스트, 로깅 등 대부분의 자동화에 사용합니다.

```json
{
  "type": "command",
  "command": "prettier --write \"$CLAUDE_FILE_PATH\"",
  "timeout": 30
}
```

#### 타입 2: HTTP 요청

외부 서버에 데이터를 보냅니다. Slack 알림, 감사 로그 전송 등에 활용합니다.

```json
{
  "type": "http",
  "url": "https://hooks.slack.com/your-webhook",
  "timeout": 10
}
```

#### 타입 3: LLM 프롬프트

다른 AI에게 "이 작업이 안전한지 확인해줘"라고 검토를 맡깁니다.

```json
{
  "type": "prompt",
  "prompt": "이 명령이 보안 위험이 있는지 확인: $ARGUMENTS",
  "model": "claude-haiku-4-5"
}
```

#### 타입 4: Agent 훅

도구 접근 권한이 있는 완전한 AI 에이전트로 실행됩니다. 파일을 읽거나 명령어를 실행해서 검증하는 복잡한 작업에 사용합니다.

```json
{
  "type": "agent",
  "prompt": "테스트가 모두 통과했는지 확인하고, 실패한 게 있으면 원인을 분석해줘.",
  "timeout": 120
}
```

> 💡 **초보자 팁**: 처음에는 **셸 명령어 타입**만 사용해도 충분합니다. 나머지 타입은 자동화에 익숙해진 후 도입하세요.

### 주요 훅 이벤트 -- 언제 실행되나?

20개 이상의 이벤트가 있지만, 초보자가 꼭 알아야 할 핵심 이벤트는 다음과 같습니다.

```
┌─────────────────────────────────────────────────┐
│              도구 생명주기 이벤트                   │
│                                                  │
│  [PreToolUse]  ──>  [도구 실행]  ──>  [PostToolUse] │
│   도구 실행 전        (Bash,         도구 실행 후    │
│   "차단 가능!"        Write 등)     "후처리 가능!"   │
└─────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────┐
│              세션 생명주기 이벤트                   │
│                                                  │
│  [SessionStart] ──> [대화 진행] ──> [Stop]        │
│   세션 시작 시        ...          Claude 턴 종료  │
│   "환경 설정!"                    "마무리 확인!"   │
└─────────────────────────────────────────────────┘
```

| 이벤트 | 언제 실행? | 활용 예시 |
|--------|----------|----------|
| `PreToolUse` | 도구 실행 **전** | 위험한 명령어 차단 |
| `PostToolUse` | 도구 실행 **후** | 파일 저장 후 자동 포매팅 |
| `Stop` | Claude 턴 종료 전 | 필수 작업 완료 확인 |
| `SessionStart` | 세션 시작 시 | 환경 변수 로드 |
| `UserPromptSubmit` | 사용자 입력 제출 시 | 입력 검증 |
| `FileChanged` | 감시 파일 변경 시 | .env 변경 감지 |

### 실제 설정 예시

#### 예시 1: 파일 저장 후 자동 포매팅 (가장 인기 있는 훅)

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "matcher": "Write",
        "hooks": [
          {
            "type": "command",
            "command": "prettier --write \"$CLAUDE_FILE_PATH\" 2>/dev/null || true"
          }
        ]
      }
    ]
  }
}
```

**해석**: Claude가 `Write` 도구로 파일을 저장한 **후에**, prettier로 자동 포매팅합니다. `2>/dev/null || true`는 prettier가 없어도 에러가 나지 않게 합니다.

#### 예시 2: 위험한 명령어 차단

```json
{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "if echo \"$CLAUDE_TOOL_INPUT\" | grep -q 'rm -rf'; then echo 'rm -rf는 허용되지 않습니다' >&2; exit 2; fi"
          }
        ]
      }
    ]
  }
}
```

**해석**: Claude가 Bash 명령어를 실행하기 **전에**, `rm -rf`가 포함되어 있으면 종료 코드 2로 차단합니다. Claude는 이 피드백을 받고 더 안전한 대안을 찾습니다.

#### 예시 3: 모든 도구 사용 로깅

```json
{
  "hooks": {
    "PostToolUse": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "echo \"$(date -u +%Y-%m-%dT%H:%M:%SZ) $CLAUDE_TOOL_NAME\" >> ~/.claude-tool-log.txt",
            "async": true
          }
        ]
      }
    ]
  }
}
```

**해석**: matcher가 없으므로 **모든** 도구 호출이 로그에 기록됩니다. `async: true`로 백그라운드 실행해서 Claude 작업을 방해하지 않습니다.

### 훅 설정 위치

```
우선순위 (낮음 → 높음):

~/.claude/settings.json          ← 사용자 설정 (모든 프로젝트 적용)
.claude/settings.json            ← 프로젝트 설정 (팀 공유용)
.claude/settings.local.json      ← 로컬 설정 (나만 사용, git 제외)
```

> 💡 **초보자 팁**: 모든 파일의 훅이 합쳐져서 실행됩니다. 높은 우선순위 파일이 낮은 우선순위를 덮어쓰는 게 아니라, **모두 함께** 실행된다는 점이 중요합니다. Claude Code에서 `/hooks` 명령어를 입력하면 대화형 설정 메뉴가 열립니다.

### 한줄 요약

> 훅은 "어떤 일이 일어나면 자동으로 이것을 해라"라는 규칙입니다. 자동 포매팅, 위험 차단, 로깅 등을 설정 한 번으로 영구 자동화할 수 있습니다.

## 3.3 스킬(Skills) -- 나만의 명령어 만들기

### 개념 설명

스킬은 **마크다운 파일로 만든 나만의 슬래시 명령어**입니다.

요리 레시피를 생각해보세요. 레시피 카드에 "재료 준비 -> 볶기 -> 양념 -> 담기" 순서를 적어두면, 누구든 그 레시피대로 요리할 수 있죠? 스킬은 Claude에게 주는 레시피 카드입니다.

```
일상 비유:                          Claude Code 스킬:

레시피 카드:                       SKILL.md:
"1. 양파 썰기                      "1. package.json 버전 업데이트
 2. 기름에 볶기                     2. CHANGELOG 작성
 3. 소금 간하기"                    3. 테스트 실행
                                    4. 커밋 & 태그"

"이 레시피대로 해줘"               "/release 1.2.3"
```

### 왜 중요한가

반복 작업을 매번 설명할 필요가 없습니다. `/release 1.2.3`처럼 한 줄로 복잡한 워크플로우를 실행할 수 있습니다. 팀원 모두가 같은 방식으로 작업할 수 있고, 마크다운이라 코딩 없이도 만들 수 있습니다.

### 스킬 만들기 -- 3단계

#### 1단계: 디렉토리 만들기

```bash
mkdir -p .claude/skills/deploy
```

#### 2단계: SKILL.md 작성하기

```markdown
---
description: 프로젝트를 스테이징 환경에 배포합니다
argument-hint: 환경이름 (staging 또는 production)
allowed-tools: Bash, Read
---

$ARGUMENTS 환경에 배포를 진행합니다.

단계:
1. 현재 브랜치가 main인지 확인
2. `npm test`로 모든 테스트 통과 확인
3. `npm run build`로 빌드
4. `./scripts/deploy.sh $ARGUMENTS` 실행
5. 배포 결과를 요약하여 보고
```

#### 3단계: 사용하기

```
/deploy staging
```

Claude가 SKILL.md를 읽고, `$ARGUMENTS` 자리에 `staging`을 넣어서 순서대로 실행합니다.

> 💡 **초보자 팁**: 스킬은 **호출할 때만 로드**됩니다(Lazy Loading). 스킬을 100개 만들어도 Claude Code 시작 속도에 영향이 없으니 마음껏 만드세요!

### SKILL.md 핵심 설정 (Frontmatter)

```yaml
---
description: 슬래시 명령어 목록에 표시되는 설명
argument-hint: 자동완성에서 보여줄 인자 힌트
allowed-tools: Bash, Write, Read    # 이 스킬이 사용할 수 있는 도구
when_to_use: Claude가 자동으로 호출할 시기 설명
model: claude-sonnet-4-6            # 이 스킬 전용 모델 (선택)
context: fork                        # 격리된 하위에이전트에서 실행 (선택)
paths: "**/*.py"                     # 이 파일 패턴 작업 시에만 활성화
---
```

### 인자 사용법

#### 기본 인자 (`$ARGUMENTS`)

```markdown
$ARGUMENTS 이름의 React 컴포넌트를 만들어주세요.
```

호출: `/new-component Button` -> `$ARGUMENTS`가 `Button`으로 대체됩니다.

#### 이름이 있는 인자

```yaml
---
arguments: [name, directory]
---

$name 컴포넌트를 $directory 폴더에 만들어주세요.
```

#### 인라인 셸 명령어

`!`을 붙인 백틱으로 호출 시점에 셸 명령어를 실행하고 결과를 삽입할 수 있습니다.

```markdown
최근 커밋 목록:

!`git log --oneline -10`

위 변경사항을 분석하고 요약해주세요.
```

### 스킬 저장 위치

```
프로젝트 스킬:                     사용자 스킬:
.claude/skills/                     ~/.claude/skills/
  deploy/                             standup/
    SKILL.md  → /deploy                SKILL.md  → /standup
  database/
    migrate/                        이 프로젝트에서만 사용          모든 프로젝트에서 사용
      SKILL.md  → /database:migrate
    seed/
      SKILL.md  → /database:seed
```

하위 디렉토리는 콜론(`:`)으로 구분됩니다. `/skills`를 입력하면 사용 가능한 모든 스킬 목록을 볼 수 있습니다.

### 조건부 스킬 -- 특정 파일 작업 시에만 활성화

```yaml
---
description: Django 모델 리뷰
paths: "**/*.py"
when_to_use: Django 모델 파일을 편집할 때 사용
---
```

Python 파일을 읽거나 쓸 때만 이 스킬이 자동으로 Claude 컨텍스트에 로드됩니다.

### 실전 예시: React 컴포넌트 생성기

```markdown
---
description: 테스트 포함 React 컴포넌트 생성
argument-hint: ComponentName (예: UserProfile)
allowed-tools: Write, Bash
---

$ARGUMENTS 이름의 React 컴포넌트를 생성합니다.

1. `src/components/$ARGUMENTS/$ARGUMENTS.tsx` 생성:
   - TypeScript 함수형 컴포넌트
   - Props 인터페이스 (`$ARGUMENTSProps`)
   - JSDoc 주석
   - Default export

2. `src/components/$ARGUMENTS/$ARGUMENTS.test.tsx` 생성:
   - React Testing Library 렌더링 테스트
   - 스냅샷 테스트

3. `src/components/$ARGUMENTS/index.ts` 생성 (re-export)

4. `npx tsc --noEmit`으로 타입 에러 확인
```

호출: `/new-component Button`

### 한줄 요약

> 스킬은 마크다운 파일로 만드는 나만의 슬래시 명령어입니다. 반복되는 복잡한 워크플로우를 한 줄 명령어로 실행할 수 있습니다.

## 3.4 MCP 서버 -- 외부 세계와 연결하기

### 개념 설명

MCP(Model Context Protocol)는 Claude Code를 외부 서비스와 연결하는 **표준 규격**입니다.

USB 포트를 생각해보세요. USB-C 하나로 모니터, 키보드, 외장하드, 충전기를 모두 연결할 수 있죠? MCP는 AI 세계의 USB-C입니다. 하나의 표준 규격으로 데이터베이스, Slack, GitHub, Jira 등 어떤 서비스든 Claude에 연결할 수 있습니다.

```
                      MCP 이전                    MCP 이후

                 ┌── 전용 플러그인 A ──┐        ┌────────────┐
  Claude Code ───┼── 전용 플러그인 B ──┼─ DB    │            │
                 ├── 전용 플러그인 C ──┤        │  MCP 표준   │── DB
                 └── 전용 플러그인 D ──┘        │  (USB-C)   │── Slack
                                                │            │── GitHub
  각 서비스마다 다른 방식                         │            │── Jira
  N개 서비스 x M개 AI = N*M 통합               └────────────┘

                                                하나의 표준으로 모두 연결
```

### 왜 중요한가

Claude Code가 혼자서는 할 수 없는 일을 할 수 있게 됩니다.
- 데이터베이스에서 직접 쿼리 실행
- Slack 메시지 읽기/보내기
- Jira 티켓 생성/업데이트
- 파일시스템의 특정 영역 접근

### 3가지 연결 방식

```
┌──────────────────────────────────────────────────────┐
│                    MCP 전송 방식                       │
│                                                       │
│  1. Stdio (로컬)      2. HTTP (원격)    3. SSE (실시간) │
│  ┌───────────────┐   ┌───────────┐    ┌───────────┐  │
│  │ Claude Code   │   │  Claude   │    │  Claude   │  │
│  │      ↕        │   │    ↕      │    │    ↕      │  │
│  │  stdin/stdout  │   │  HTTPS   │    │   SSE    │  │
│  │      ↕        │   │    ↕      │    │    ↕      │  │
│  │ 로컬 프로세스  │   │ 원격 서버 │    │ 이벤트서버│  │
│  └───────────────┘   └───────────┘    └───────────┘  │
│                                                       │
│  가장 흔함             API 서비스용       실시간 스트림  │
└──────────────────────────────────────────────────────┘
```

### MCP 서버 추가하기

#### 방법 1: CLI 명령어 (가장 쉬움)

```bash
# 파일시스템 서버 추가
claude mcp add filesystem -- npx -y @modelcontextprotocol/server-filesystem /tmp

# 프로젝트 팀 공유용
claude mcp add --scope project filesystem -- npx -y @modelcontextprotocol/server-filesystem /tmp

# 나만 사용 (글로벌)
claude mcp add --scope user my-db -- npx -y @my-org/mcp-server-postgres
```

#### 방법 2: 설정 파일 (`.mcp.json`)

```json
{
  "mcpServers": {
    "filesystem": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-filesystem", "/home/user/projects"],
      "env": {
        "NODE_ENV": "production"
      }
    },
    "my-api": {
      "type": "http",
      "url": "https://mcp.example.com/v1",
      "headers": {
        "Authorization": "Bearer $MY_API_TOKEN"
      }
    }
  }
}
```

### 설정 스코프와 우선순위

| 스코프 | 위치 | 용도 | 우선순위 |
|--------|------|------|---------|
| local | `.claude/settings.local.json` | 나만의 오버라이드 | 높음 |
| project | `.mcp.json` (프로젝트 루트) | 팀 공유 | 중간 |
| user | `~/.claude.json` | 모든 프로젝트 | 낮음 |

> 💡 **초보자 팁**: 같은 이름의 서버가 여러 스코프에 있으면 **local이 우선**합니다. 팀 설정을 내 환경에 맞게 오버라이드할 때 유용합니다.

### MCP 서버 관리하기

```bash
/mcp                          # 전체 서버 상태 확인
/mcp enable my-database       # 특정 서버 활성화
/mcp disable analytics        # 특정 서버 비활성화
/mcp reconnect filesystem     # 연결 끊긴 서버 재연결
```

**서버 상태 한눈에 보기:**

| 상태 | 의미 | 대응 |
|------|------|------|
| connected | 정상 작동 중 | 없음 |
| pending | 초기화 중 | 잠시 기다림 |
| failed | 연결 실패 | 명령어/환경변수 확인 |
| needs-auth | 인증 필요 | OAuth 인증 완료 |
| disabled | 의도적 비활성 | 필요 시 enable |

### 실전 예시: PostgreSQL 연결

```json
{
  "mcpServers": {
    "postgres": {
      "command": "npx",
      "args": ["-y", "@modelcontextprotocol/server-postgres"],
      "env": {
        "POSTGRES_CONNECTION_STRING": "$DATABASE_URL"
      }
    }
  }
}
```

```bash
# 사용 전에 환경 변수 설정
export DATABASE_URL="postgresql://user:pass@localhost:5432/mydb"
claude    # 이제 Claude가 DB를 직접 쿼리할 수 있음
```

### 🤔 MCP 트러블슈팅

| 문제 | 원인 | 해결 |
|------|------|------|
| Failed 상태 | 명령어 없음 | `which npx` 확인 |
| 도구가 안 보임 | 인증 미완료 | `/mcp`에서 needs-auth 확인 |
| 환경변수 미적용 | 변수 미설정 | `echo $YOUR_VAR`로 확인 |
| Windows에서 실패 | npx 경로 문제 | `cmd /c` 래퍼 사용 |

### 도구 호출 시 권한 확인

MCP 도구는 실행 전에 반드시 사용자 승인을 받습니다.

```
Claude: "mcp__postgres__query 도구를 실행하려고 합니다."
        입력: SELECT * FROM users LIMIT 10

  [Allow once]  [Allow always]  [Deny]
   한 번 허용      항상 허용       차단
```

### 한줄 요약

> MCP는 AI 세계의 USB-C 포트입니다. 하나의 표준으로 데이터베이스, API, 파일시스템 등 어떤 외부 서비스든 Claude Code에 연결할 수 있습니다.

## 3.5 권한 시스템 -- AI의 안전벨트

### 개념 설명

권한 시스템은 Claude Code가 **어떤 작업을 자동으로 하고, 어떤 작업은 허락을 받을지** 정하는 안전장치입니다.

건물 출입 시스템을 생각해보세요. 사원증이 있으면 일반 사무실은 자유롭게 들어가지만, 서버실은 별도 승인이 필요하고, 금고실은 절대 혼자 들어갈 수 없죠.

```
건물 출입 시스템:                   Claude Code 권한 시스템:

일반 사무실 → 자유 출입            파일 읽기 → 자동 허용
회의실 → 예약 필요                셸 명령어 → 승인 필요
서버실 → 관리자 승인 필요         위험한 명령 → 차단
금고실 → 절대 불가                .git/ 수정 → 항상 차단
```

### 왜 중요한가

AI가 강력한 만큼 위험도 따릅니다. 실수로 중요한 파일을 삭제하거나, 민감한 정보를 외부로 보내는 일을 방지해야 합니다. OWASP과 같은 보안 기관에서도 AI 에이전트에 대해 **최소 권한 원칙(Least Privilege)**을 강조합니다. 필요한 최소한의 권한만 부여하고, 필요한 순간에만 권한을 활성화하는 것이 핵심입니다.

### 5가지 권한 모드

```
       안전                                              빠름
  ◀───────────────────────────────────────────────────────▶

  plan        default       acceptEdits     dontAsk   bypass
  (읽기전용)   (기본)        (편집 자동)    (규칙만)   (전체해제)

  탐색용       일상용        신뢰하는         자동화     CI/CD
              (추천!)       프로젝트        스크립트   컨테이너만
```

| 모드 | 파일 읽기 | 파일 편집 | 셸 명령어 | 추천 상황 |
|------|:---------:|:---------:|:---------:|----------|
| `plan` | 자동 | 차단 | 차단 | 새 코드베이스 탐색 |
| `default` | 자동 | 승인 필요 | 승인 필요 | **일상 사용 (추천)** |
| `acceptEdits` | 자동 | 자동 | 승인 필요 | 신뢰하는 프로젝트 |
| `dontAsk` | 자동 | 규칙에 따라 | 규칙에 따라 | 스크립트/자동화 |
| `bypassPermissions` | 자동 | 자동 | 자동 | Docker/CI만 사용! |

### 권한 모드 설정 방법 3가지

```bash
# 1. CLI 플래그로 (일회성)
claude --permission-mode acceptEdits

# 2. 세션 중에 변경
/permissions    # 대화형 메뉴에서 선택

# 3. settings.json에 기본값 설정 (영구)
{
  "defaultPermissionMode": "acceptEdits"
}
```

> 💡 **초보자 팁**: 처음에는 `default` 모드로 시작하세요. Claude가 뭘 하려는지 하나하나 확인하면서 패턴을 이해한 후, 자주 허용하는 작업들을 규칙으로 추가하면 됩니다.

### Allow/Deny 규칙 -- 세밀한 제어

매번 "허용"을 누르는 게 귀찮다면, 규칙을 만들어 자동화할 수 있습니다.

```json
{
  "permissions": {
    "allow": [
      "Bash(git status)",
      "Bash(git diff *)",
      "Bash(git log *)",
      "Bash(npm run *)",
      "Read(*)"
    ],
    "deny": [
      "Bash(rm -rf *)",
      "Bash(sudo *)",
      "Bash(curl * | bash)",
      "Edit(.env*)",
      "Write(~/.ssh/**)"
    ]
  }
}
```

**패턴 해석:**

| 패턴 | 의미 |
|------|------|
| `git status` | 정확히 `git status`만 허용 |
| `git *` | `git`으로 시작하는 모든 명령어 허용 |
| `Read(*)` | 모든 파일 읽기 허용 |
| `Edit(.env*)` | .env 파일 편집 차단 |
| `mcp__myserver` | MCP 서버의 모든 도구 차단 |
| `mcp__myserver__read_db` | 특정 MCP 도구만 허용/차단 |

### 권한 결정 파이프라인 (7단계)

Claude가 어떤 도구를 실행하려 할 때, 다음 순서로 권한을 확인합니다.

```
                    도구 실행 요청
                         │
                         v
              ┌─── 1. Deny 규칙 ───┐
              │    매칭되면 즉시 차단  │
              └────────┬───────────┘
                       │ 통과
                       v
              ┌─── 2. Ask 규칙 ────┐
              │   매칭되면 확인 요청  │
              └────────┬───────────┘
                       │ 통과
                       v
              ┌─── 3. 도구 자체 권한 ─┐
              │   도구별 기본 안전 수준 │
              └────────┬───────────┘
                       │ 통과
                       v
              ┌─── 4. 안전 확인 ────┐
              │   .git/ .claude/ 등  │
              │   → 항상 확인 요청   │
              └────────┬───────────┘
                       │ 통과
                       v
              ┌─── 5. 모드 확인 ────┐
              │  현재 권한 모드 적용  │
              └────────┬───────────┘
                       │ 통과
                       v
              ┌─── 6. Allow 규칙 ──┐
              │   매칭되면 즉시 허용  │
              └────────┬───────────┘
                       │ 통과
                       v
              ┌─── 7. 기본 동작 ───┐
              │   사용자에게 질문    │
              └───────────────────┘
```

**핵심 포인트**: Deny가 **가장 먼저**, Allow가 **거의 마지막**에 평가됩니다. 이것은 **보안 우선(deny-first)** 설계입니다.

### 항상 차단되는 작업 (어떤 모드든)

권한 모드와 관계없이 **절대** 허용되지 않는 작업들이 있습니다.

- `.claude/`, `.git/` 설정 디렉토리 수정
- `.bashrc`, `.zshrc` 등 셸 설정 파일 수정
- 경로 제한 우회 시도

### 복합 명령어 처리

`&&`, `||`, `;`, `|`로 연결된 명령어는 **각각 독립적으로** 검사됩니다.

```bash
git status && rm -rf /     # git status는 OK, rm -rf는 차단
                            # → 전체 명령어 차단 (가장 제한적 결과 적용)
```

### 한줄 요약

> 권한 시스템은 AI의 안전벨트입니다. 5가지 모드와 Allow/Deny 규칙으로 "자동 실행 / 허락 받기 / 절대 불가"를 세밀하게 제어합니다.

{

스킬 추가 (`.claude/skills/new-component/SKILL.md`):

## 3.7 자주 묻는 질문

### 🤔 훅 vs 스킬, 뭘 써야 하나요?

| 상황 | 선택 | 이유 |
|------|------|------|
| "파일 저장할 때마다 포매팅" | 훅 | **매번 자동** 실행해야 하니까 |
| "릴리스 프로세스 실행해줘" | 스킬 | **필요할 때만** 실행하니까 |
| "위험한 명령어를 막고 싶어" | 훅 | **항상** 감시해야 하니까 |
| "스탠드업 보고서 작성해줘" | 스킬 | **요청할 때만** 필요하니까 |

**한마디 공식: "매번 자동으로" -> 훅, "필요할 때만" -> 스킬**

### 🤔 MCP 서버가 꼭 필요한가요?

아닙니다! MCP 없이도 Claude Code는 충분히 강력합니다. 다만 데이터베이스 쿼리, 외부 API 호출 등 **Claude가 직접 접근할 수 없는 외부 서비스**와 연동할 때 필요합니다.

### 🤔 bypassPermissions 모드는 언제 쓰나요?

**Docker 컨테이너나 CI 환경처럼 격리된 환경에서만** 사용하세요. 로컬 개발 환경에서 사용하면 Claude가 실수로 중요한 시스템 파일을 수정할 수 있습니다.

### 🤔 훅에서 에러가 나면 어떻게 되나요?

종료 코드가 0이나 2가 아닌 다른 값이면, 에러 메시지가 사용자에게 표시되지만 **Claude의 작업은 계속 진행**됩니다. Claude는 이 에러를 알지 못합니다.

### 🤔 여러 설정 파일에 훅을 넣으면 어떻게 되나요?

모든 설정 파일의 훅이 **합쳐져서 모두 실행**됩니다. 높은 우선순위 파일이 낮은 것을 덮어쓰지 않습니다.

## 3.8 핵심 포인트 정리

### 📌 훅 (Hooks)

- **정의**: "이벤트가 발생하면 자동으로 실행되는 규칙"
- **핵심**: 종료 코드 0(성공), 2(차단/피드백), 기타(에러)
- **4가지 타입**: 셸 명령어, HTTP 요청, LLM 프롬프트, Agent
- **실무 필수**: `PostToolUse`(자동 포매팅) + `PreToolUse`(위험 차단)

### 📌 스킬 (Skills)

- **정의**: "마크다운으로 만든 재사용 가능한 슬래시 명령어"
- **핵심**: SKILL.md 파일 + `$ARGUMENTS` 치환 + Lazy Loading
- **저장소**: `.claude/skills/`(프로젝트) 또는 `~/.claude/skills/`(전역)
- **실무 팁**: 반복 워크플로우는 무조건 스킬로 만들자

### 📌 MCP 서버

- **정의**: "AI 세계의 USB-C -- 외부 서비스를 표준 규격으로 연결"
- **핵심**: Stdio(로컬)/HTTP(원격)/SSE(실시간) 3가지 전송 방식
- **관리**: `/mcp` 명령어로 활성화/비활성화/재연결
- **보안**: 모든 MCP 도구 호출은 사전 승인 필요

### 📌 권한 시스템

- **정의**: "AI가 할 수 있는 일과 없는 일을 제어하는 안전장치"
- **핵심**: 5가지 모드 + Allow/Deny 규칙 + 7단계 결정 파이프라인
- **원칙**: Deny-first (거부가 항상 먼저 평가)
- **실무 팁**: `default` 모드로 시작, 자주 허용하는 작업을 규칙에 추가

### 📌 조합 공식

```
훅 (자동 강제)  +  스킬 (온디맨드)  +  MCP (외부 연결)
                    │
                    ↓
              권한 시스템 (안전장치)
```
