## 1.1 Claude Code란 무엇인가?

### 한 줄 정의

**Claude Code는 터미널에서 실행되는 AI 코딩 에이전트입니다.**

"잠깐, 그냥 AI 챗봇 아닌가요?" 라고 생각하실 수 있습니다. 아닙니다. 완전히 다릅니다.

### 비유로 이해하기

일반 AI 챗봇(ChatGPT, 웹 Claude 등)은 **상담 창구의 안내원**과 같습니다.
질문하면 답변해주지만, 직접 뭔가를 해주지는 않습니다.

> "이 서류 어떻게 작성하나요?" → "이렇게 작성하시면 됩니다" (설명만 해줌)

반면 Claude Code는 **숙련된 개발자 동료**와 같습니다.
여러분의 컴퓨터에서 직접 파일을 읽고, 코드를 수정하고, 명령어를 실행합니다.

> "이 버그 좀 고쳐줘" → 파일을 열어보고, 문제를 찾고, 코드를 수정하고, 테스트까지 돌림

### Claude Code가 할 수 있는 것들

```
파일 읽기/편집     →  소스 코드를 읽고, 직접 수정하고, 변경사항을 보여줌
셸 명령 실행       →  테스트 돌리기, 빌드하기, git 작업 등을 직접 실행
코드베이스 검색     →  프로젝트 전체에서 패턴을 찾아냄
웹 검색            →  문서, API 스펙을 터미널에서 바로 조회
다중 에이전트       →  복잡한 작업을 여러 에이전트가 나눠서 병렬 처리
MCP 서버 연결      →  데이터베이스, API, 내부 도구 확장 가능
```

### 왜 중요한가?

기존에는 AI에게 "이렇게 하면 돼"라는 답을 듣고, **직접 손으로** 코드를 작성해야 했습니다.
Claude Code는 답을 듣는 단계를 넘어서 **직접 실행까지** 해줍니다.
이것이 "에이전트(Agent)"라는 이름이 붙은 이유입니다.

> 💡 **초보자 팁**
> Claude Code를 처음 쓴다면, 터미널에서 `claude`를 실행하고
> "이 프로젝트의 구조를 설명해줘"라고 말해보세요.
> Claude가 직접 파일을 탐색하면서 설명해주는 것을 볼 수 있습니다.

**한줄 요약**: Claude Code는 단순한 챗봇이 아니라, 여러분의 터미널에서 직접 코드를 읽고 쓰고 실행하는 AI 개발자 동료입니다.

## 1.2 에이전틱 루프: 핵심 개념

### "에이전틱 루프"가 뭔가요?

이름이 어렵게 느껴질 수 있지만, 개념은 단순합니다.

**에이전틱 루프(Agentic Loop)** = AI가 **스스로 판단하고 → 행동하고 → 결과를 보고 → 다시 판단하는** 반복 과정

### 비유: 요리사의 레시피 실행

요리사가 복잡한 요리를 만드는 과정을 생각해보세요.

```
1. 레시피를 읽는다 (지시사항 이해)
2. 필요한 재료가 있는지 냉장고를 확인한다 (상황 파악)
3. 양파를 썬다 (행동)
4. 잘 썰어졌는지 확인한다 (결과 관찰)
5. "좋아, 다음은 볶기" (다음 행동 결정)
6. 프라이팬에 기름을 두르고 볶는다 (행동)
7. 색이 나는지 확인한다 (결과 관찰)
8. ... 요리가 완성될 때까지 반복
```

Claude Code도 정확히 이렇게 동작합니다.

```
1. 사용자의 요청을 읽는다           → "로그인 버그를 고쳐줘"
2. 프로젝트 상황을 파악한다          → 어떤 파일이 있는지, git 상태는 어떤지
3. 파일을 읽어본다                  → login.ts를 열어서 코드 확인
4. 문제를 발견한다                  → "아, JWT 만료 검증이 빠져있구나"
5. 코드를 수정한다                  → validateToken() 함수 수정
6. 테스트를 돌려본다                → bun test 실행
7. 결과를 확인한다                  → "테스트 통과! 수정 완료"
8. 사용자에게 결과를 보고한다
```

이 **"판단 → 행동 → 관찰 → 반복"** 사이클이 바로 에이전틱 루프입니다.

### 핵심 특징: "도구 호출이 없을 때까지 반복"

에이전틱 루프는 **더 이상 할 일이 없을 때** 자동으로 멈춥니다.
요리사가 모든 재료를 손질하고 요리를 완성하면 자연스럽게 멈추는 것처럼요.

### 왜 "루프(Loop)"인가?

한 번의 질문-답변으로 끝나지 않고, **여러 단계를 순환(loop)하면서** 작업을 완료하기 때문입니다.

```
  ┌─→ 판단(추론) ──→ 행동(도구 사용) ──→ 관찰(결과 확인) ─┐
  │                                                      │
  └──────────── 더 할 일이 있으면 다시 ←──────────────────┘
                     ↓ (할 일 없으면)
                   최종 응답
```

> 💡 **초보자 팁**
> 에이전틱 루프를 한마디로? **"AI가 알아서 여러 단계를 돌면서 일을 끝내는 구조"**입니다.
> 기존 챗봇이 "1타 1피"라면, 에이전트는 "셀프 멀티태스킹"입니다.

**한줄 요약**: 에이전틱 루프는 AI가 스스로 생각하고, 도구를 쓰고, 결과를 확인하며 작업을 반복하는 자율적 순환 구조입니다.

## 1.3 챗봇 vs 에이전트: 근본적인 차이

### 가장 쉬운 구분법

| | 챗봇 (ChatGPT 등) | 에이전트 (Claude Code) |
|---|---|---|
| **동작 방식** | 질문 1개 → 답변 1개 | 요청 1개 → 여러 단계 자동 실행 |
| **비유** | 백과사전 | 숙련된 조수 |
| **API 호출** | 1회 | 여러 회 (루프) |
| **도구 사용** | 못함 | 파일, 터미널, 웹 등 직접 사용 |
| **실행 환경** | 웹 브라우저 (원격 서버) | 내 컴퓨터 터미널 (로컬) |
| **중간 수정** | 불가 | 중간에 방향 수정 가능 |

### 비유: 내비게이션 vs 전문 기사

**챗봇** = 내비게이션
- "강남역까지 어떻게 가나요?" → 경로를 알려줌
- 직접 운전해주지는 않음
- 중간에 길이 막혀도 알아서 대응 못함

**에이전트** = 전문 기사가 있는 자율주행차
- "강남역까지 가주세요" → 직접 운전함
- 중간에 길이 막히면 알아서 우회
- 도착하면 "도착했습니다"라고 알려줌

### 기술적으로 보면

```
[챗봇]
사용자 → "JWT 버그 고쳐줘" → LLM 1회 호출 → "이렇게 고치세요" (끝)

[에이전트 = Claude Code]
사용자 → "JWT 버그 고쳐줘"
  → LLM 호출 #1: "먼저 관련 파일을 찾아보겠습니다" → Grep 도구 실행
  → LLM 호출 #2: "이 파일을 읽어보겠습니다" → Read 도구 실행
  → LLM 호출 #3: "문제를 찾았습니다. 수정합니다" → Edit 도구 실행
  → LLM 호출 #4: "테스트를 실행합니다" → Bash 도구 실행
  → LLM 호출 #5: "모든 테스트 통과. 수정 완료." (끝)
```

핵심 차이는 단 하나입니다:
**챗봇은 LLM을 1회 호출하고, 에이전트는 LLM을 루프 안에서 여러 번 호출합니다.**

> 💡 **초보자 팁**
> Claude Code가 작업하는 걸 보면 터미널에 여러 단계가 쭉 나옵니다.
> "파일 읽는 중...", "코드 수정 중...", "테스트 실행 중..."
> 이게 다 에이전틱 루프가 돌고 있는 것입니다.

**한줄 요약**: 챗봇은 "한 번 물어보면 한 번 답하는" 구조이고, 에이전트는 "한 번 시키면 알아서 여러 단계를 돌며 끝내는" 구조입니다.

## 1.4 6단계 순환 프로세스 완전 해부

이제 Claude Code의 에이전틱 루프가 구체적으로 어떻게 돌아가는지 6단계로 살펴보겠습니다.

### 전체 아키텍처 (큰 그림)

```
┌──────────────────────────────────────────────────────────────┐
│                    Claude Code 프로세스                        │
│               (여러분의 컴퓨터에서 실행됩니다!)                    │
│                                                              │
│  ┌───────────┐    ┌──────────────┐    ┌────────────────┐     │
│  │ 1단계      │    │ 2단계         │    │ 3단계           │     │
│  │ 사용자가   │───>│ 상황 파악     │───>│ Claude가        │     │
│  │ 요청 입력  │    │ (컨텍스트 조립)│    │ 생각하고 도구선택│     │
│  └───────────┘    └──────────────┘    └──────┬─────────┘     │
│       ^                                       │              │
│       │                                       v              │
│  ┌────┴──────┐    ┌──────────────┐    ┌────────────────┐     │
│  │ 6단계      │    │ 5단계         │    │ 4단계           │     │
│  │ 더 할 일   │<───│ 도구 실행     │<───│ 권한 확인       │     │
│  │ 있나 확인  │    │ 결과 받기     │    │ "실행해도 될까?" │     │
│  └───────────┘    └──────────────┘    └────────────────┘     │
│                                                              │
│  ※ 3→4→5→6→3 을 할 일이 없을 때까지 반복합니다                   │
│  ※ 원격 서버 없이, 여러분 컴퓨터에서만 실행됩니다                  │
└──────────────────────────────────────────────────────────────┘
```

### 각 단계 자세히 살펴보기

---

#### 1단계: 사용자 메시지 전송 - "무엇을 해야 하는지 알려주기"

**비유**: 식당에서 주문하는 것과 같습니다.

여러분이 터미널에서 Claude Code에게 요청을 입력합니다.

```bash
# 대화형 모드 (직접 입력)
claude
> "로그인 기능에 입력값 검증을 추가해줘"

# 비대화형 모드 (스크립트에서 사용)
claude -p "이 코드베이스의 구조를 설명해줘"
```

**두 가지 모드**:
- **대화형**: 터미널에서 직접 대화 (일반적인 사용)
- **비대화형**: CI/CD 파이프라인이나 스크립트에서 자동화용

---

#### 2단계: 컨텍스트 조립 - "현재 상황 파악하기"

**비유**: 의사가 진료 전에 환자의 차트(병력, 알레르기, 현재 복용약)를 확인하는 것과 같습니다.

Claude Code는 여러분의 요청을 처리하기 전에, 먼저 "지금 상황이 어떤지" 파악합니다.

```
수집하는 정보들:
├── 🗂️ Git 상태       →  현재 브랜치, 최근 커밋 5개, 변경된 파일
├── 📋 CLAUDE.md      →  프로젝트 규칙 (빌드 명령어, 코딩 컨벤션 등)
├── 📅 오늘 날짜       →  "Today's date is 2026-03-31"
├── 🔧 사용 가능한 도구 →  Read, Edit, Bash, Grep 등
└── 👤 사용자 설정      →  개인 선호도, 응답 언어 등
```

이 모든 정보가 하나의 "시스템 프롬프트"로 합쳐져서 Claude에게 전달됩니다.

> 💡 **초보자 팁**
> CLAUDE.md 파일이 없어도 Claude Code는 동작합니다.
> 하지만 CLAUDE.md가 있으면 프로젝트에 맞는 훨씬 정확한 답을 줍니다.
> 처음 시작할 때 `/init` 명령어를 입력하면 자동으로 만들어줍니다.

---

#### 3단계: Claude의 추론 및 도구 선택 - "어떻게 할지 생각하기"

**비유**: 셰프가 주문서를 보고 "어떤 재료를 먼저 손질할지" 계획을 세우는 것과 같습니다.

Claude는 수집된 컨텍스트와 여러분의 요청을 Anthropic API로 보내고, Claude 모델이 "다음에 무엇을 해야 할지" 판단합니다.

```
사용자 요청: "로그인 폼에 입력값 검증 추가해줘"

Claude의 사고 과정:
"로그인 폼이 어디 있는지 먼저 찾아야겠다"
 → 결정: Grep 도구로 "login" 관련 파일 검색
```

이때 Claude는 **도구 호출(tool_use) 블록**을 생성합니다. 이것은 쉽게 말해 "이 도구를 이런 방식으로 사용하겠다"는 계획서입니다.

---

#### 4단계: 권한 확인 - "이거 실행해도 되나요?"

**비유**: 회사에서 큰 결재가 필요한 업무는 상사의 승인을 받는 것과 같습니다.

Claude Code는 모든 도구 호출 전에 "이걸 실행해도 될까?" 확인합니다.

```
권한 확인 흐름:

파일 읽기 (Read, Grep, Glob)  →  항상 자동 승인 ✅ (안전하니까)
파일 수정 (Edit, Write)       →  모드에 따라 다름
셸 명령 실행 (Bash)           →  대부분 확인 요청 ⚠️ (위험할 수 있으니까)
```

**권한 모드 3가지**:

| 모드 | 동작 | 비유 |
|------|------|------|
| `default` | 파일 수정, 명령 실행 전 매번 물어봄 | 신입사원 (다 물어봄) |
| `acceptEdits` | 파일 수정은 자동, 명령 실행만 물어봄 | 경력사원 (코드 수정은 알아서) |
| `bypassPermissions` | 모든 것을 자동 실행 | 대표이사 (전결권) |

> 💡 **초보자 팁**
> 처음에는 기본 모드(`default`)로 사용하세요.
> Claude가 뭘 하려는지 하나하나 확인하면서 배울 수 있습니다.
> 익숙해지면 `acceptEdits`로 바꾸면 됩니다.
> `bypassPermissions`는 자동화(CI/CD)에서만 사용하세요.

---

#### 5단계: 도구 실행 및 결과 반환 - "실제로 일하기"

**비유**: 셰프가 계획대로 재료를 손질하고 조리하는 단계입니다.

승인된 도구가 실행되고, 그 결과가 대화에 추가됩니다.

```
도구 실행 예시:

Grep("login" 검색) → 결과: "src/components/LoginForm.tsx에서 발견"
    ↓
Read(LoginForm.tsx) → 결과: 파일 내용 전체
    ↓
Edit(검증 코드 추가) → 결과: "파일 수정 완료"
    ↓
Bash("bun test") → 결과: "Tests passed: 42/42"
```

Claude Code에서 사용 가능한 도구들:

```
도구 분류:
├── 📄 파일 도구
│   ├── Read     — 파일 읽기 (코드, 이미지, PDF까지!)
│   ├── Edit     — 코드 수정 (정확한 위치를 찾아서 교체)
│   ├── Write    — 새 파일 만들기
│   └── Glob     — 파일 이름 패턴으로 검색 (*.tsx 등)
│
├── 💻 셸 도구
│   └── Bash     — 터미널 명령 실행 (빌드, 테스트, git 등)
│
├── 🔍 검색 도구
│   └── Grep     — 코드 내용 검색 (정규식 지원)
│
├── 🌐 웹 도구
│   ├── WebFetch — 웹페이지 내용 가져오기
│   └── WebSearch— 웹 검색
│
└── 🤖 에이전트 도구
    └── Task     — 하위 에이전트 생성 (복잡한 작업 분할)
```

---

#### 6단계: 루프 판단 - "더 할 일이 있나?"

**비유**: 요리사가 "다음 단계가 있나?" 레시피를 확인하는 것과 같습니다.

Claude가 결과를 보고 판단합니다:

```
할 일이 더 있다면?  →  3단계로 돌아가서 다음 도구 호출
                         (파일 찾기 → 읽기 → 수정 → 테스트 순으로 계속)

할 일이 끝났다면?  →  최종 텍스트 응답 생성하고 종료
                         "로그인 폼에 이메일/비밀번호 검증을 추가했습니다.
                          테스트도 모두 통과합니다."
```

### 루프가 멈추는 5가지 조건

```
정상 종료:
  ① 할 일이 다 끝남 (도구 호출 없이 텍스트만 응답)

비정상 종료:
  ② 턴 예산 초과 — "너무 많은 단계를 돌았어요" (무한 루프 방지)
  ③ 비용 예산 초과 — "정해진 금액을 넘겼어요" (돈 관리)
  ④ 재시도 한도 초과 — "같은 실패를 너무 많이 반복했어요"
  ⑤ 사용자가 직접 중단 — Ctrl+C 등으로 멈춤
```

> 💡 **초보자 팁**
> CI/CD에서 Claude Code를 쓸 때는 반드시 `--max-budget-usd` 옵션을 설정하세요.
> 예: `claude -p "코드 리뷰" --max-budget-usd 0.50`
> 이렇게 하면 무한 루프로 인한 비용 폭탄을 방지할 수 있습니다.

**한줄 요약**: 6단계 루프는 "요청받기 → 상황파악 → 생각 → 허가 → 실행 → 반복판단"의 순환이며, 할 일이 끝나면 자동으로 멈춥니다.

## 1.5 쿼리 엔진: Claude Code의 심장부

### 쿼리 엔진이 뭔가요?

에이전틱 루프의 각 "턴(turn)"을 실제로 돌리는 엔진입니다.
심장이 피를 온몸에 보내듯, 쿼리 엔진이 루프 전체를 구동합니다.

**비유**: 쿼리 엔진은 **오케스트라 지휘자**와 같습니다.
연주자(도구)들이 언제 무엇을 연주할지 조율하고, 전체 공연(작업)이 매끄럽게 흘러가도록 관리합니다.

### 쿼리 엔진의 5가지 역할

```
┌─────────────────────────────────────────────────────┐
│                   쿼리 엔진 (query.ts)               │
│                                                     │
│  ① 스트리밍        Claude의 답변을 실시간으로 터미널에  │
│                    글자 단위로 보여줌                   │
│                                                     │
│  ② 도구 디스패칭    "어떤 도구를 써야 하지?" →            │
│                    올바른 도구 핸들러로 연결             │
│                                                     │
│  ③ 예산 관리        턴 수(API 호출 횟수)와              │
│                    비용(USD)을 추적하며 한도 관리        │
│                                                     │
│  ④ 결과 크기 관리   도구 결과가 너무 크면?               │
│                    → 임시 파일로 저장하고 요약만 전달     │
│                                                     │
│  ⑤ 컴팩션 트리거    대화가 너무 길어지면?                │
│                    → 오래된 대화를 요약해서 공간 확보     │
│                                                     │
└─────────────────────────────────────────────────────┘
```

### 결과 크기 관리: 왜 필요한가?

**비유**: 보고서를 쓸 때, 1,000페이지짜리 원본 자료를 그대로 첨부하면 아무도 안 읽습니다.
핵심만 요약하고, 원본은 "별첨"으로 두는 게 효율적이죠.

Claude Code도 마찬가지입니다. 도구 실행 결과가 너무 크면:

```
결과 생성 → 크기 확인 → 한도(maxResultSizeChars) 초과?
                              │
                    ┌─────────┴─────────┐
                    │ 아니오             │ 예
                    v                   v
              결과 그대로 전달      임시 파일에 저장하고
                                  미리보기만 Claude에게 전달
```

이렇게 하면 Claude의 "작업 공간"(컨텍스트 윈도우)이 넘치지 않습니다.

### 컴팩션: 장시간 대화 관리

**비유**: 아주 긴 회의를 하면, 초반에 했던 이야기는 기억이 희미해지죠.
회의록 담당자가 앞부분을 요약해놓으면 효율적입니다.

```
대화 진행 →→→→→→→→→→→→→→→→→→→→→→→→→→→→→→→

컨텍스트 윈도우 사용량: [████████████████░░░░] 80% ← 거의 다 참!
                                                  │
                            ┌─────────────────────┘
                            v
                    컴팩션(Compaction) 트리거!
                    - 오래된 대화 → 핵심만 요약
                    - 최근 대화 → 그대로 유지
                    - 원본은 디스크에 안전하게 보관
```

> 💡 **초보자 팁**
> 장시간 대화에서 "아까 말한 거 기억해?"라고 물었는데 못 알아듣는다면,
> 컴팩션 때문에 이전 내용이 요약되었을 수 있습니다.
> 중요한 내용은 **최근 메시지에서 다시 한번 언급**하거나,
> 아예 `CLAUDE.md`에 적어두면 항상 기억합니다.
> `/compact` 명령으로 수동으로 컴팩션할 수도 있습니다.

**한줄 요약**: 쿼리 엔진은 스트리밍, 도구 연결, 예산 관리, 결과 압축, 대화 요약이라는 5가지 핵심 역할로 에이전틱 루프 전체를 구동하는 심장부입니다.

## 1.6 하위 에이전트: 일을 나눠서 시키기

### 하위 에이전트(Sub-agent)란?

**비유**: 팀장이 큰 프로젝트를 받으면, 팀원들에게 일을 나눠서 시키죠.
Claude Code도 똑같습니다. 복잡한 작업을 받으면 "하위 에이전트(Sub-agent)"를 만들어서 일을 분담합니다.

```
┌────────────────────────────────────────────────┐
│              팀장 (부모 에이전트)                  │
│          "전체 프로젝트를 검증해줘"                 │
│                                                │
│  일을 나눠서 시킵니다:                             │
│                                                │
│  ┌──────────────┐    ┌──────────────┐          │
│  │  팀원 A       │    │  팀원 B       │          │
│  │  "테스트 실행" │    │  "타입 체크"   │          │
│  │              │    │              │          │
│  │  독립적 작업   │    │  독립적 작업   │          │
│  │  공간          │    │  공간         │          │
│  │              │    │              │          │
│  │  결과 보고 ───┼────┼──→ 팀장에게   │          │
│  └──────────────┘    └──────────────┘          │
│                                                │
│  팀장: "테스트 통과, 타입 체크 통과. 모두 완료!"    │
└────────────────────────────────────────────────┘
```

### 왜 하위 에이전트가 필요한가?

1. **병렬 처리**: 테스트 + 타입 체크 + 린트를 동시에 돌릴 수 있음
2. **컨텍스트 분리**: 각 팀원(에이전트)이 자기 작업만 집중 → 메인 컨텍스트 깨끗하게 유지
3. **비용 절약**: 탐색 작업은 저렴한 모델(haiku)에 맡기고, 핵심 작업만 고급 모델(opus)이 처리

### 포그라운드 vs 백그라운드

```
포그라운드 실행:                 백그라운드 실행:
팀장: "이 일 먼저 해"           팀장: "이 일 하고 있어, 나는 다른 거 할게"
팀장: (기다리는 중...)           팀장: (다른 일 하는 중...)
팀원: "다 했습니다!"            팀원: "다 했습니다!" (나중에 알림)
팀장: "좋아, 다음 일 하자"      팀장: "오, 끝났구나. 결과 확인해볼게"
```

### 중요한 특징: 컨텍스트 격리

하위 에이전트는 부모의 대화 내용을 **자동으로 물려받지 않습니다**.
마치 새로운 팀원이 투입되면 따로 브리핑을 해줘야 하는 것처럼요.

```
부모 에이전트의 머릿속            하위 에이전트의 머릿속
┌─────────────────┐            ┌──────────────────┐
│ 30턴의 대화 기록  │            │                  │
│ 여러 파일 분석    │  ──X──>   │ 전달받은 지시사항만  │
│ 중간 추론들       │  (상속X)   │ 알고 있음          │
│ 이전 도구 결과    │            │                  │
└─────────────────┘            └──────────────────┘
```

따라서 하위 에이전트에게 일을 시킬 때는, **필요한 맥락을 명확하게 전달**해야 합니다.

**좋은 예시**:
```
"src/auth/login.ts의 42번째 줄에서 JWT 만료 검증이 빠져있습니다.
 validateToken() 함수에 exp 클레임 확인을 추가하고,
 tests/auth.test.ts를 실행해서 통과하는지 확인해주세요."
```

**나쁜 예시**:
```
"아까 그 버그 좀 고쳐줘"
(→ 하위 에이전트는 "아까 그 버그"가 뭔지 모릅니다!)
```

### 하위 에이전트를 쓰면 안 되는 경우

| 이런 작업은... | 이 도구를 쓰세요 |
|---|---|
| 파일 하나 읽기 | Read (하위 에이전트는 과함) |
| 코드에서 뭔가 검색 | Grep |
| 단순 명령 하나 실행 | Bash |
| 2~3개 파일 확인 | Read를 여러번 |

> 💡 **초보자 팁**
> 하위 에이전트(Task)는 오버헤드가 큽니다.
> 새로운 컨텍스트 생성, 시스템 프롬프트 구성, API 호출 등이 필요합니다.
> **5개 이상의 도구 호출이 필요한 복합 작업**에만 사용하세요.
> 간단한 작업에 Task를 쓰면 시간과 비용이 낭비됩니다.

**한줄 요약**: 하위 에이전트는 복잡한 작업을 나눠서 병렬 처리하는 메커니즘이며, 각자 독립된 작업 공간에서 실행되므로 필요한 맥락을 명확히 전달해야 합니다.

## 1.7 대화형 vs 비대화형 모드

Claude Code는 두 가지 방식으로 사용할 수 있습니다.

### 대화형(REPL) 모드 - 일상적인 개발

**비유**: 동료와 대화하면서 같이 일하는 것

```bash
# 대화 시작
claude

# 이후 자유롭게 대화
> "이 프로젝트 구조 설명해줘"
> "UserService 클래스에 테스트 추가해줘"
> "방금 수정한 것 되돌려줘"
```

특징:
- 실시간으로 Claude의 답변이 터미널에 스트리밍
- 도구 실행 전 승인/거절 가능
- 이전 대화를 기억하고 이어서 작업 가능

### 비대화형(Print) 모드 - 자동화/스크립트

**비유**: 메모를 써서 건네주고 결과만 받는 것

```bash
# 한 번 실행하고 결과만 받기
claude -p "이 코드베이스의 TODO 코멘트를 모두 찾아줘"

# CI/CD 파이프라인에서 코드 리뷰
echo "이 PR의 변경사항을 리뷰해줘" | claude --output-format json --print
```

특징:
- UI 렌더링 없음 (서버/CI 환경에 적합)
- 결과를 stdout으로 출력 (다른 도구와 연결 가능)
- `--max-budget-usd`로 비용 제한 필수

**한줄 요약**: 대화형은 일상 개발용, 비대화형은 자동화/CI 파이프라인용이며, 상황에 맞게 선택하면 됩니다.

## 1.8 자주 묻는 질문

### 🤔 자주 묻는 질문

**Q1. Claude Code는 인터넷에 내 코드를 보내나요?**

네, 여러분의 코드가 Anthropic API로 전송됩니다. 하지만 Claude Code 자체는 **완전히 로컬에서 실행**됩니다. 원격 실행 서버가 없으며, 모든 도구 실행은 여러분의 컴퓨터에서 일어납니다. API로 전송되는 것은 Claude가 "생각"하기 위한 대화 내용뿐입니다.

**Q2. 에이전틱 루프가 무한히 돌 수 있나요?**

이론적으로는 가능하지만, 실제로는 **여러 안전장치**가 있습니다:
- 턴 예산: 최대 몇 번까지 반복할지 제한
- 비용 예산: 최대 몇 달러까지 쓸지 제한 (`--max-budget-usd`)
- 사용자 중단: 언제든 Ctrl+C로 멈출 수 있음

**Q3. Claude Code가 위험한 명령을 실행할 수도 있나요?**

기본 모드에서는 파일 수정이나 셸 명령 실행 전에 **반드시 사용자 확인**을 받습니다.
`rm -rf /` 같은 위험한 명령도 여러분이 "허용"을 누르지 않으면 실행되지 않습니다.
추가로 권한 설정에서 특정 명령을 완전히 차단(deny)할 수도 있습니다.

**Q4. 하위 에이전트를 너무 많이 만들면 어떻게 되나요?**

비용이 늘어납니다. 각 하위 에이전트는 독립적인 API 호출을 하므로, 불필요하게 많이 만들면 비용이 급증할 수 있습니다. 결과 크기도 100,000자로 제한되며, 재귀적으로 계속 하위 에이전트를 만드는 것도 방지됩니다(평면 구조).

**Q5. 대화를 이어서 할 수 있나요?**

네! 대화는 자동으로 `~/.claude/` 폴더에 저장됩니다.
`claude --resume`으로 이전 대화를 이어서 할 수 있습니다.
특정 세션을 재개하려면 `claude --resume <session-id>`를 사용하세요.

**Q6. Claude Code를 설치하려면 뭐가 필요한가요?**

Node.js 18 이상과 npm만 있으면 됩니다.
```bash
npm install -g @anthropic-ai/claude-code
claude    # 첫 실행 시 브라우저에서 로그인
```

## 1.9 핵심 포인트 정리

### 📌 핵심 포인트

```
┌─────────────────────────────────────────────────────────┐
│                                                         │
│  1. Claude Code는 챗봇이 아니라 "에이전트"이다            │
│     → 직접 파일을 읽고, 코드를 수정하고, 명령을 실행한다    │
│                                                         │
│  2. 에이전틱 루프 = 판단 → 행동 → 관찰 → 반복             │
│     → 할 일이 없을 때까지 자동으로 순환한다                 │
│                                                         │
│  3. 6단계: 입력 → 컨텍스트 → 추론 → 권한 → 실행 → 반복    │
│     → 모든 것이 여러분의 로컬 터미널에서 실행된다           │
│                                                         │
│  4. 쿼리 엔진이 루프 전체를 구동한다                       │
│     → 스트리밍, 도구 연결, 예산 관리, 컴팩션까지           │
│                                                         │
│  5. 하위 에이전트로 복잡한 작업을 병렬 분할할 수 있다       │
│     → 각자 독립 컨텍스트, 충분한 맥락 전달이 핵심          │
│                                                         │
│  6. 권한 시스템이 안전을 보장한다                          │
│     → 위험한 작업은 반드시 사용자 확인을 거친다             │
│                                                         │
└─────────────────────────────────────────────────────────┘
```
