## 2.1 컨텍스트 윈도우: AI의 작업 메모리

### "컨텍스트 윈도우"가 뭔가요?

컨텍스트 윈도우는 Claude가 **한 번에 볼 수 있는 정보의 총량**입니다.

### 비유: 책상 위의 공간

여러분의 책상을 상상해보세요. 책상이 넓을수록 더 많은 자료를 동시에 펼쳐놓고 일할 수 있지만, 아무리 넓어도 한계가 있죠. 꽉 차면 오래된 자료부터 치워야 합니다.

**컨텍스트 윈도우 = Claude의 책상 크기**

Claude의 "책상" 위에는 이런 것들이 올라갑니다:

```
컨텍스트 윈도우에 담기는 것들:
├── 📋 CLAUDE.md (프로젝트 매뉴얼)       ← 항상 올라가 있음
├── 🔧 시스템 프롬프트 (도구 목록 등)     ← 항상 올라가 있음
├── 💬 지금까지의 대화 기록              ← 계속 쌓임
├── 📄 읽은 파일 내용                   ← 도구 결과로 추가됨
├── 💻 실행한 명령의 출력               ← 도구 결과로 추가됨
└── 🔍 검색 결과                       ← 도구 결과로 추가됨
```

### 토큰: 컨텍스트의 측정 단위

컨텍스트 윈도우의 크기는 **토큰(token)** 단위로 측정됩니다.

```
토큰이란?
  영어: 1 토큰 ≈ 0.75 단어 (= 3~4글자). "Hello world" = 약 2 토큰
  한국어: 1 토큰 ≈ 1~2 음절. "안녕하세요" = 약 3~4 토큰 (영어보다 토큰을 더 많이 씀)

Claude의 컨텍스트 윈도우: 약 200,000 토큰 (200K)
  = 영문 소설 2~3권 분량 / A4 약 500장
  ⚠️ 넉넉해 보이지만, 큰 파일 몇 개 읽으면 금방 50% 이상 참!
```

### 왜 중요한가?

컨텍스트 윈도우가 꽉 차면 두 가지 일이 벌어집니다:

1. **컴팩션 발생**: 오래된 대화가 요약됨 → 세부 정보 손실
2. **성능 저하**: 정보가 너무 많으면 Claude가 중요한 부분을 놓칠 수 있음

> 💡 **초보자 팁**
> 컨텍스트 윈도우를 효율적으로 쓰는 간단한 방법들:
> - 큰 파일을 읽을 때는 필요한 부분만 읽기 (offset/limit 사용)
> - 검색 결과 수를 제한하기
> - 독립적인 작업은 하위 에이전트로 분리하기
> - 대화가 길어지면 `/compact` 명령으로 수동 정리

**한줄 요약**: 컨텍스트 윈도우는 Claude의 "작업 책상"이며, CLAUDE.md, 대화 기록, 도구 결과가 모두 이 안에 들어가야 하므로 효율적 관리가 핵심입니다.

## 2.2 CLAUDE.md: Claude Code의 업무 매뉴얼

### CLAUDE.md가 뭔가요?

**비유**: 새로 온 동료에게 주는 **업무 매뉴얼**입니다.

신입사원이 첫 출근했을 때, "우리 팀은 이렇게 일해" "이 프로젝트는 이런 구조야" "빌드할 때는 이 명령어 써"라고 알려주는 문서를 상상해보세요. CLAUDE.md가 바로 그 역할입니다.

```
CLAUDE.md 없이 시작하면:
사용자: "테스트 돌려줘"
Claude: "어떤 테스트 프레임워크를 쓰시나요? 명령어가 뭔가요?"
→ 매번 물어봐야 함 😩

CLAUDE.md가 있으면:
사용자: "테스트 돌려줘"
Claude: (CLAUDE.md에서 "테스트: bun test"를 이미 알고 있음)
        → 바로 bun test 실행! 🎉
```

### CLAUDE.md에 뭘 넣어야 하나요?

```markdown
# 좋은 CLAUDE.md 예시

## 명령어
- 빌드: `pnpm run build`
- 테스트: `pnpm test`
- 린트: `pnpm run lint --fix`

## 아키텍처
- 모노레포 (pnpm workspace)
- packages/core: 비즈니스 로직
- packages/api: REST API (Express)

## 코딩 규칙
- 2칸 들여쓰기, camelCase/PascalCase
- any 금지 → unknown + 타입 가드
- console.log 금지 → logger 사용
```

```
넣으면 안 되는 것:
  ❌ API 키나 비밀번호 (보안 위험!)
  ❌ 너무 긴 문서 (컨텍스트 낭비)
  ❌ 매일 바뀌는 정보 (메모이제이션 때문에 세션 중 반영 안 됨)
  ❌ 모호한 지시 ("깨끗한 코드 작성해줘")
```

### CLAUDE.md 만드는 방법

```bash
# 방법 1: 자동 생성 (추천)
claude
> /init

# 방법 2: 직접 생성
# 프로젝트 루트에 CLAUDE.md 파일을 만들면 됩니다
```

> 💡 **초보자 팁**
> `/init` 명령어를 쓰면 Claude가 프로젝트를 분석해서 CLAUDE.md를 자동으로 만들어줍니다.
> 만들어진 파일을 검토하고 필요한 부분을 수정한 뒤 Git에 커밋하세요.
> 초보자가 범하는 가장 큰 실수는 **CLAUDE.md를 아예 만들지 않는 것**입니다.
> CLAUDE.md 없이 시작하면, 매 세션마다 "콜드 스타트" 상태가 됩니다.

**한줄 요약**: CLAUDE.md는 Claude에게 "이 프로젝트에서는 이렇게 일해"라고 알려주는 업무 매뉴얼이며, 없으면 매번 처음부터 설명해야 합니다.

## 2.3 4계층 메모리 구조 완전 정복

### 왜 계층이 필요한가?

**비유**: 회사의 규칙 체계를 생각해보세요.

```
회사 전체 규칙    →  "출퇴근 시간은 9시~6시" (모든 직원에게 적용)
부서 규칙         →  "코드 리뷰 필수" (개발팀에만 적용)
팀 규칙           →  "PR은 2명 이상 승인" (우리 팀에만 적용)
개인 설정         →  "모니터 밝기 80%" (나에게만 적용)
```

Claude Code의 메모리도 비슷한 계층 구조입니다.
범위가 넓을수록 기본 규칙이고, 좁을수록 세부 설정입니다.

### 4계층 한눈에 보기

```
┌────────────────────────────────────────────────────────────┐
│                                                            │
│   1단계: 관리형 (Managed)     ← 회사 전체 규칙              │
│   /etc/claude-code/CLAUDE.md   "조직의 보안 정책"           │
│   ⚠️ 절대 재정의 불가                                       │
│                                                            │
│   ────────────────────────────────────────────────────      │
│                                                            │
│   2단계: 사용자 (User)        ← 내 개인 글로벌 설정          │
│   ~/.claude/CLAUDE.md          "응답은 한국어로"             │
│   📌 모든 프로젝트에 적용                                    │
│                                                            │
│   ────────────────────────────────────────────────────      │
│                                                            │
│   3단계: 프로젝트 (Project)   ← 팀 공유 규칙                 │
│   ./CLAUDE.md                  "빌드: pnpm build"           │
│   📌 Git에 커밋, 팀원 모두 공유                              │
│                                                            │
│   ────────────────────────────────────────────────────      │
│                                                            │
│   4단계: 로컬 (Local)         ← 나만의 프로젝트 설정          │
│   ./CLAUDE.local.md            "내 DB 포트: 5433"           │
│   📌 .gitignore에 추가, 나만 사용                            │
│                                                            │
│                                                            │
│   우선순위:  로컬 > 프로젝트 > 사용자 > 관리형                │
│   (단, 관리형의 "정책"은 항상 최우선)                         │
│                                                            │
└────────────────────────────────────────────────────────────┘
```

### 각 계층 자세히 알아보기

| 단계 | 경로 | 비유 | 핵심 |
|------|------|------|------|
| **1. 관리형** | `/etc/claude-code/CLAUDE.md` | 국가 법률 | 재정의 불가, IT 관리자 설정 |
| **2. 사용자** | `~/.claude/CLAUDE.md` | 개인 사무실 룰 | 모든 프로젝트에 적용, Git 비포함 |
| **3. 프로젝트** | `./CLAUDE.md` | 팀 컨벤션 문서 | Git 커밋, 팀 전체 공유 |
| **4. 로컬** | `./CLAUDE.local.md` | 책상 서랍 메모장 | .gitignore 필수, 나만 사용 |

**사용자 메모리 예시** (`~/.claude/CLAUDE.md`):
```markdown
- 응답은 항상 한국어로
- 커밋 메시지는 Conventional Commits 형식
- 2칸 들여쓰기 사용
```

**로컬 메모리 예시** (`CLAUDE.local.md`):
```markdown
- 내 로컬 DB 포트: 5433 (기본 5432가 아님)
- API 서버는 localhost:8081에서 실행 중
```

**프로젝트 메모리 파일 탐색 방식**: CWD부터 루트(/)까지 올라가며 CLAUDE.md를 찾고, 발견된 모든 파일이 합쳐집니다. CWD에 가까운 파일이 더 높은 우선순위를 가집니다.

### 어떤 계층에 무엇을 넣어야 할까?

| 정보 | 넣을 곳 | 이유 |
|------|---------|------|
| "응답은 한국어로" | 사용자 (2단계) | 모든 프로젝트에 적용되는 개인 선호 |
| "빌드: pnpm build" | 프로젝트 (3단계) | 팀원 모두가 알아야 하는 정보 |
| "내 DB 포트: 5433" | 로컬 (4단계) | 내 환경에만 해당 |
| "프로덕션 접근 금지" | 관리형 (1단계) | 조직 전체 보안 정책 |
| API 키, 비밀번호 | 어디에도 넣지 마세요! | 보안 위험 |

> 💡 **초보자 팁**
> 처음에는 **프로젝트 CLAUDE.md 하나만** 만들어도 충분합니다.
> 프로젝트 규모가 커지고 팀원이 늘어나면 다른 계층을 추가하세요.
> 시작이 중요하지, 완벽한 구조가 중요한 게 아닙니다.

**한줄 요약**: 4계층 메모리는 관리형(조직) → 사용자(개인 글로벌) → 프로젝트(팀 공유) → 로컬(개인 비공개) 순서이며, CWD에 가까울수록 우선순위가 높습니다.

## 2.4 @include 디렉티브: 메모리를 모듈화하기

### CLAUDE.md가 너무 길어지면?

프로젝트가 커지면 CLAUDE.md에 넣어야 할 내용도 많아집니다.
수백 줄짜리 CLAUDE.md 하나보다, **여러 파일로 나눠서 관리**하는 게 효율적입니다.

**비유**: 백과사전을 한 권으로 만들지 않고, 여러 권으로 나누는 것과 같습니다.
의학편, 과학편, 역사편... 필요한 권만 꺼내 볼 수 있죠.

### @include 사용법

CLAUDE.md 안에서 `@` 기호로 다른 파일을 참조할 수 있습니다.

```markdown
# 프로젝트 CLAUDE.md

## 공통 규칙
@./docs/conventions/typescript.md
@./docs/conventions/testing.md
@./docs/architecture.md

## 패키지별 안내
@./packages/api/CLAUDE.md
@./packages/web/CLAUDE.md

커밋 전 `pnpm test && pnpm typecheck` 실행 필수
```

이렇게 하면 각 참조 파일의 내용이 자동으로 CLAUDE.md에 합쳐집니다.

### @include 구문과 규칙

| 구문 | 의미 |
|------|------|
| `@filename` | CLAUDE.md 기준 상대 경로 |
| `@./relative/path` | 명시적 상대 경로 |
| `@~/home/path` | 홈 디렉토리 기준 |
| `@/absolute/path` | 절대 경로 |

주요 규칙: 코드 블록 안의 `@`는 무시됨 / 순환 참조 자동 차단 / 없는 파일은 에러 없이 무시 / 최대 5단계 중첩 / 텍스트 파일만 지원 / 프로젝트 외부 파일은 승인 필요

> 💡 **초보자 팁**
> @include의 가장 좋은 점은 **이미 있는 문서를 재사용**할 수 있다는 것입니다.
> docs/ 폴더에 이미 아키텍처 문서가 있다면, 따로 CLAUDE.md에 복사하지 말고
> `@./docs/architecture.md`로 참조하세요. 중복이 사라집니다.

**한줄 요약**: @include는 CLAUDE.md가 길어질 때 여러 파일로 나눠 관리하는 방법이며, 기존 문서를 재사용할 수 있어 중복 작성이 불필요합니다.

## 2.5 Rules 파일: 상황에 맞는 규칙 적용

### 문제: 모든 규칙이 항상 로드되면?

프로젝트에 API 규칙, 프론트엔드 규칙, 테스트 규칙이 모두 있다고 합시다.
API 코드를 수정할 때 프론트엔드 규칙까지 로드되면 **컨텍스트 낭비**입니다.

### 해결: .claude/rules/ 디렉토리

**비유**: 회사 매뉴얼을 챕터별로 나누는 것입니다.
영업팀 매뉴얼, 개발팀 매뉴얼, HR 매뉴얼을 따로 두고 필요할 때만 참조하는 방식이죠.

```
프로젝트/
├── CLAUDE.md                    # 핵심만 간략하게
└── .claude/
    └── rules/
        ├── api-design.md        # API 작업할 때만 적용
        ├── frontend-style.md    # 프론트엔드 작업할 때만 적용
        ├── testing.md           # 테스트 작성할 때만 적용
        └── git-workflow.md      # 항상 적용 (경로 지정 없음)
```

### Frontmatter로 경로 타겟팅

Rules 파일에 YAML frontmatter를 넣으면, **특정 파일을 작업할 때만** 해당 규칙이 로드됩니다.

```markdown
---
paths:
  - "src/api/**"
  - "src/services/**"
---

## API/서비스 규칙
- 의존성 주입 항상 사용
- 구현체 직접 임포트 금지
```

이 규칙은 `src/api/` 또는 `src/services/` 하위 파일을 작업할 때만 컨텍스트에 포함됩니다.
Glob 패턴 예: `src/api/**` (하위 전체), `**/*.test.ts` (모든 테스트 파일)

**경로 타겟팅의 효과**: API 작업 시 프론트엔드 규칙까지 로드되면 컨텍스트 낭비! 타겟팅하면 필요한 규칙만 로드되어 더 많은 코드와 대화를 담을 수 있습니다.

> 💡 **초보자 팁**
> 프로젝트가 작다면 rules 파일을 나눌 필요 없습니다.
> CLAUDE.md 하나로 충분합니다.
> 프로젝트가 커지고, 규칙이 많아지고, 컨텍스트가 부족해질 때 도입하세요.

**한줄 요약**: Rules 파일은 규칙을 모듈화하고 경로 타겟팅으로 필요한 규칙만 로드하여 컨텍스트를 절약하는 방법입니다.

## 2.6 메모이제이션: 똑같은 일을 두 번 하지 않기

### 메모이제이션이 뭔가요?

**비유**: 전화번호부를 매번 처음부터 찾지 않고, 자주 쓰는 번호를 **즐겨찾기**에 저장해두는 것입니다.

Claude Code는 세션이 시작될 때 메모리 파일을 찾고, 시스템 정보를 수집합니다.
이 결과를 **캐시(cache)**해두고, 같은 세션 동안은 다시 계산하지 않습니다.

### 어떻게 동작하나요?

```
[세션 시작] → Git 상태 수집 + CLAUDE.md 탐색 → [캐시에 저장]
                                                      │
[이후 매 턴마다] → 캐시에서 즉시 반환 (파일 시스템 재탐색 X)
```

**왜 필요한가?** 에이전틱 루프는 여러 턴을 돕니다. 메모이제이션 없이 20턴을 돌면 매번 파일 탐색을 하므로 10초를 낭비합니다. 캐시를 쓰면 첫 턴(0.5초)에만 탐색하고 나머지는 즉시 반환됩니다.

### 주의할 점: 세션 중 CLAUDE.md 수정

메모이제이션 때문에, **진행 중인 세션에서 CLAUDE.md를 수정해도 즉시 반영되지 않습니다**.

```
상황: 세션 진행 중에 CLAUDE.md를 수정함

❌ 기대: "바로 적용되겠지?"
✅ 현실: "이 세션에서는 캐시된 이전 버전이 계속 사용됨"

해결 방법:
방법 1: 새 세션을 시작한다 (claude 명령 재실행)
방법 2: claude --resume 으로 세션을 재개한다 (CLAUDE.md 재발견됨)
방법 3: /memory 명령어로 메모리 에디터를 열어 편집한다 (저장 시 리로드)
```

> 💡 **초보자 팁**
> 세션 중에 CLAUDE.md를 바꿔야 한다면, `/memory` 명령어를 사용하세요.
> 이 명령어로 수정하면 자동으로 리로드됩니다.
> 또는 대화로 "CLAUDE.md에 ~ 규칙 추가해줘"라고 요청하면
> Claude가 적절한 파일을 찾아서 직접 수정해줍니다.

**한줄 요약**: 메모이제이션은 세션 내에서 컨텍스트 수집 결과를 캐시하여 성능을 높이는 최적화이며, 세션 중 CLAUDE.md 변경 시 /memory 또는 세션 재시작이 필요합니다.

## 2.7 컴팩션: 긴 대화를 요약해서 공간 확보

### 컴팩션이 뭔가요?

**비유**: 장시간 회의를 하면 화이트보드가 꽉 차죠. 그러면 앞부분을 사진으로 찍어두고(원본 보존), 핵심만 남기고 지워서(요약) 새로운 내용을 쓸 공간을 만듭니다.

Claude Code의 컴팩션도 정확히 이렇게 동작합니다.

```
┌──────────────────────────────────────────────────────┐
│                    컨텍스트 윈도우                       │
│                                                      │
│  [컴팩션 전]                                           │
│  ┌────────────────────────────────────────────────┐   │
│  │ CLAUDE.md  │  대화 1~20턴  │  대화 21~30턴     │   │
│  │ (항상 유지)  │  (매우 길다)  │  (최근 대화)       │   │
│  └────────────────────────────────────────────────┘   │
│  사용량: ████████████████████░ 95% ← 거의 꽉 참!       │
│                                                      │
│  [컴팩션 후]                                           │
│  ┌────────────────────────────────────────────────┐   │
│  │ CLAUDE.md  │ 요약  │  대화 21~30턴             │   │
│  │ (항상 유지)  │(짧음) │  (최근 대화, 그대로 유지)    │   │
│  └────────────────────────────────────────────────┘   │
│  사용량: ████████░░░░░░░░░░░░ 40% ← 공간 확보!         │
│                                                      │
│  ⚡ 원본은 디스크(~/.claude/)에 안전하게 보존됨          │
└──────────────────────────────────────────────────────┘
```

### 자동 컴팩션 vs 수동 컴팩션

| | 자동 컴팩션 | 수동 컴팩션 |
|---|---|---|
| **트리거** | 컨텍스트 95% 도달 시 자동 | `/compact` 명령 입력 |
| **장점** | 신경 쓸 필요 없음 | 타이밍 제어 가능 |
| **단점** | 예상치 못한 시점에 발생 | 직접 관리해야 함 |

### 컴팩션에서 보존되는 것과 사라지는 것

```
항상 보존:
  ✅ CLAUDE.md 내용
  ✅ 최근 대화 (몇 턴)
  ✅ 시스템 프롬프트 (도구 목록 등)

요약되는 것 (세부 내용 손실):
  ⚠️ 오래된 대화 내용
  ⚠️ 이전 도구 실행 결과의 세부 사항
  ⚠️ 초반 대화에서의 구체적 코드 조각

완전히 보존되는 곳:
  💾 디스크의 원본 트랜스크립트 (컴팩션은 API 전송 내용만 영향)
```

### 컴팩션 대비 전략

| 전략 | 방법 |
|------|------|
| 최근 메시지에서 반복 언급 | "아까 말한 validateUserInput 함수 말인데..." → 컴팩션 대상 아님 |
| CLAUDE.md에 핵심 규칙 기록 | 컨텍스트에서 절대 제거되지 않음 |
| 하위 에이전트로 작업 분리 | 메인 컨텍스트 오염 방지 |
| `/compact`으로 수동 컴팩션 | 큰 작업 시작 전에 정리 |

관련 설정: `DISABLE_AUTO_COMPACT=1` (자동 끄기, 주의 필요), `CLAUDE_CODE_MAX_CONTEXT_TOKENS` (크기 직접 설정), 훅 `PreCompact`/`PostCompact` (전/후 커스텀 로직)

> 💡 **초보자 팁**
> 컴팩션은 Claude Code가 알아서 해주는 기능이니 너무 걱정하지 마세요.
> 다만 대화가 길어질수록 초반 내용이 "흐려진다"는 점만 기억하세요.
> 정말 중요한 정보는 CLAUDE.md에 적어두면 대화가 아무리 길어져도 유지됩니다.

**한줄 요약**: 컴팩션은 긴 대화에서 오래된 내용을 요약하여 컨텍스트 공간을 확보하는 자동 메커니즘이며, 중요한 정보는 CLAUDE.md에 넣어야 영구 보존됩니다.

## 2.8 환경 변수: 동작을 세밀하게 제어하기

### 환경 변수로 뭘 할 수 있나요?

**비유**: 자동차의 설정 패널과 같습니다.
에어컨 온도, 사이드미러 각도, 시트 높이 등을 조절하듯이,
Claude Code의 다양한 동작을 환경 변수로 조절할 수 있습니다.

### 초보자가 알아야 할 핵심 환경 변수

| 중요도 | 변수 | 용도 |
|--------|------|------|
| 꼭 알기 | `ANTHROPIC_API_KEY` | API 키 인증 (OAuth 대신 사용할 때) |
| 꼭 알기 | `ANTHROPIC_MODEL` | 기본 모델 변경 |
| 꼭 알기 | `DISABLE_AUTO_COMPACT=1` | 자동 컨텍스트 압축 끄기 |
| 유용함 | `CLAUDE_CODE_REMOTE=1` | 원격/CI 환경 (타임아웃 확장) |
| 유용함 | `CLAUDE_CODE_MAX_OUTPUT_TOKENS` | 응답 길이 제한 (비용 관리) |
| 고급 | `CLAUDE_CODE_USE_BEDROCK` | AWS Bedrock 사용 |

### settings.json으로 환경 변수 고정하기

매번 `export`하지 않고 `~/.claude/settings.json`의 `env` 필드에 넣으면 자동 적용:

```json
{
  "env": {
    "DISABLE_AUTO_COMPACT": "1",
    "BASH_MAX_OUTPUT_LENGTH": "30000"
  }
}
```

> 💡 **초보자 팁**
> 환경 변수를 처음부터 다 설정할 필요 없습니다.
> 기본값으로 충분히 잘 동작합니다.
> 특정 문제가 생겼을 때(비용이 너무 높다, 응답이 너무 길다 등)
> 해당 환경 변수를 찾아서 조절하세요.

**한줄 요약**: 환경 변수는 Claude Code의 세부 동작을 조절하는 설정 스위치이며, 초보자는 기본값으로 시작하고 필요할 때만 조절하면 됩니다.

## 2.9 실전 설정 가이드

### Step-by-Step: 프로젝트 처음 설정하기

```
Step 1: Claude Code 설치 및 인증

─────────────────────────────
$ cd my-project

Step 3: CLAUDE.md 자동 생성
─────────────────────────────
$ claude
> /init                  # Claude가 프로젝트를 분석해서 자동 생성

Step 4: 생성된 CLAUDE.md 검토 및 수정
─────────────────────────────
# 빌드 명령어가 맞는지, 규칙이 적절한지 확인
# 필요하면 수정

Step 5: .gitignore 설정
─────────────────────────────
# 아래 내용을 .gitignore에 추가
CLAUDE.local.md
.claude/settings.local.json

Step 6: Git에 커밋
─────────────────────────────
$ git add CLAUDE.md
$ git commit -m "docs: add CLAUDE.md for Claude Code configuration"
```

### 설정 체크리스트

```
[ ] CLAUDE.md 생성 (/init 또는 수동)
[ ] CLAUDE.md 내용 검토 (빌드 명령어, 규칙 등)
[ ] .gitignore에 CLAUDE.local.md 추가
[ ] CLAUDE.md를 Git에 커밋
[ ] (선택) ~/.claude/CLAUDE.md로 개인 글로벌 설정 (한국어 응답, 커밋 스타일 등)
[ ] (선택) CLAUDE.local.md로 로컬 환경 설정
[ ] (선택) .claude/rules/ 디렉토리로 규칙 모듈화
```

### 프로젝트 규모별 추천 설정

```
[ 소규모 프로젝트 (1인 개발) ]
├── CLAUDE.md               # 이것만 있으면 됨
└── (끝!)

[ 중규모 프로젝트 (2~5명 팀) ]
├── CLAUDE.md               # 팀 공유 규칙

└── .gitignore              # CLAUDE.local.md 제외

[ 대규모 프로젝트 (모노레포, 10명+) ]
├── CLAUDE.md               # 핵심 규칙 + @include

├── .claude/
│   ├── settings.json       # 팀 공유 설정
│   ├── settings.local.json # 개인 설정 재정의
│   └── rules/
│       ├── api-rules.md    # paths: ["src/api/**"]
│       ├── frontend.md     # paths: ["src/web/**"]
│       └── testing.md      # paths: ["**/*.test.*"]
├── docs/
│   └── architecture.md     # @include로 참조
└── .gitignore              # 로컬 파일 제외
```

**한줄 요약**: 소규모는 CLAUDE.md 하나면 충분하고, 프로젝트가 커질수록 계층 분리, Rules 모듈화, @include를 점진적으로 도입하세요.

## 2.10 자주 묻는 질문

### 🤔 자주 묻는 질문

**Q1. CLAUDE.md의 최대 크기는?**

권장 최대 크기는 **40,000자**입니다 (`MAX_MEMORY_CHARACTER_COUNT`).
이를 초과하면 경고가 표시되고, 전체 내용이 읽히지 않을 수 있습니다.
40,000자가 넘으면 `@include`로 파일을 분리하거나, `.claude/rules/`로 모듈화하세요.

**Q2. CLAUDE.md를 수정했는데 왜 반영이 안 되나요?**

메모이제이션 때문입니다. 진행 중인 세션에서는 캐시된 이전 버전이 사용됩니다.

해결 방법:
- `/memory` 명령으로 수정하기 (자동 리로드)
- 새 세션을 시작하기 (`claude` 재실행)
- `claude --resume`으로 세션 재개하기 (CLAUDE.md 재발견)

**Q3. CLAUDE.md와 CLAUDE.local.md의 차이가 뭔가요?**

| | CLAUDE.md | CLAUDE.local.md |
|---|---|---|
| **공유** | Git에 커밋 → 팀 공유 | .gitignore → 나만 사용 |
| **내용** | 팀 전체 규칙 | 내 개인 환경 설정 |
| **예시** | "테스트: bun test" | "내 DB 포트: 5433" |

**Q4. 컨텍스트 윈도우가 꽉 차면 어떻게 되나요?**

자동 컴팩션이 발동됩니다. 오래된 대화가 요약되고, 공간이 확보됩니다.
만약 `DISABLE_AUTO_COMPACT=1`을 설정해뒀다면 에러가 발생할 수 있으니 주의하세요.

**Q5. 여러 프로젝트에서 같은 설정을 쓰고 싶어요.**

`~/.claude/CLAUDE.md` (사용자 레벨 메모리)에 넣으세요.
이 파일은 모든 프로젝트에 자동으로 적용됩니다.

**Q6. CLAUDE.md를 만들지 않아도 되나요?**

네, 없어도 동작합니다. 하지만 **강력히 권장**합니다.
CLAUDE.md가 없으면 매 세션마다 Claude가 프로젝트를 처음부터 파악해야 합니다.
빌드 명령어, 테스트 방법, 코딩 규칙을 매번 알려줘야 하는 셈입니다.

**Q7. settings.json은 CLAUDE.md와 뭐가 다른가요?**

| | CLAUDE.md | settings.json |
|---|---|---|
| **용도** | Claude에게 주는 지시사항 | Claude Code 도구의 동작 설정 |
| **형식** | Markdown (자연어) | JSON (구조화된 데이터) |
| **내용** | "테스트 먼저 돌려" | `{"permissions": {"allow": ["Read"]}}` |
| **비유** | 업무 매뉴얼 | 시스템 환경설정 |

**Q8. 컴팩션을 완전히 끌 수 있나요?**

`DISABLE_AUTO_COMPACT=1`로 끌 수 있지만, **권장하지 않습니다**.
컨텍스트 윈도우가 꽉 차면 에러가 발생하고, 세션이 중단될 수 있습니다.
대신 `/compact` 명령으로 수동 컴팩션 타이밍을 제어하는 것을 추천합니다.

## 2.11 핵심 포인트 정리

### 📌 핵심 포인트

```
┌──────────────────────────────────────────────────────────┐
│                                                          │
│  1. 컨텍스트 윈도우 = Claude의 "작업 책상"                 │
│     → 모든 정보(대화, 파일, 결과)가 이 안에 들어가야 함      │
│     → 넘치면 컴팩션으로 오래된 내용 요약                     │
│                                                          │
│  2. CLAUDE.md = Claude에게 주는 "업무 매뉴얼"              │
│     → 빌드 명령어, 코딩 규칙, 아키텍처 설명                  │
│     → 없으면 매번 콜드 스타트, 있으면 즉시 생산적             │
│                                                          │
│  3. 4계층 구조: 관리형 → 사용자 → 프로젝트 → 로컬           │
│     → CWD에 가까울수록 우선순위 높음                         │
│     → 관리형의 "정책"은 절대 재정의 불가                     │
│                                                          │
│  4. @include로 메모리 모듈화 가능                           │
│     → 기존 docs/ 문서를 재사용                              │
│     → 최대 5단계 중첩, 순환 참조 자동 차단                    │
│                                                          │
│  5. 메모이제이션 = 세션 내 캐시                              │
│     → 성능 최적화를 위해 컨텍스트를 캐시함                    │
│     → 세션 중 CLAUDE.md 변경 시 /memory 또는 세션 재시작 필요 │
│                                                          │
│  6. 컴팩션 = 긴 대화의 자동 정리                            │
│     → 오래된 내용 요약, 최근 내용 보존                       │
│     → 중요한 정보는 CLAUDE.md에 넣어야 항상 유지             │
│                                                          │
│  7. 환경 변수 = 세부 동작 조절 스위치                        │
│     → 초보자는 기본값으로 시작, 필요할 때 조절                │
│     → settings.json의 env 필드로 영구 설정 가능              │
│                                                          │
└──────────────────────────────────────────────────────────┘
```
